# MP4 コンテナのバイナリ構造 詳解

Phase 2 補足。MP4（ISO Base Media File Format / ISOBMFF）の内部構造を掘り下げる。
`faststart` の意味や fMP4 との違いを理解するための土台。

---

## 1. Box（atom）— MP4の基本単位

MP4ファイルは**Box（旧称atom）**の入れ子構造でできている。すべてのデータはBoxに格納される。

### Boxのバイナリレイアウト

```
┌──────────────┬──────────────┬─────────────────┐
│  size (4B)   │  type (4B)   │  data (可変長)   │
└──────────────┴──────────────┴─────────────────┘
```

- **size**: このBox全体のバイト数（ヘッダ含む）
- **type**: 4文字のASCII識別子（`ftyp`, `moov`, `mdat` など）
- **data**: Box固有のデータ。子Boxを含むこともある

### コンテナBox と リーフBox

Boxには2種類ある。**すべてのBoxが入れ子になっているわけではない。**

**コンテナBox** — 中身が子Boxの集まり:
```
moov（コンテナBox）:
moov
├─ mvhd    ← 子Box
├─ trak    ← 子Box
│   ├─ tkhd  ← 孫Box
│   └─ mdia  ← 孫Box
└─ trak
```

**リーフBox** — 中身が固定レイアウトのバイナリデータ（子Boxを持たない）:
```
ftyp（リーフBox）:
┌──────────┬──────────┬────────────────┬───────────────┬─────────────────────┐
│ size(4B) │type(4B)  │major_brand(4B) │minor_ver(4B)  │compatible_brands(可変)│
└──────────┴──────────┴────────────────┴───────────────┴─────────────────────┘
 ← Boxヘッダ →← 子Boxではなく、ただのバイト列が直接並ぶ →
```

主なBoxの分類:

| Box | 種別 | 中身 |
|-----|------|------|
| moov, trak, mdia, minf, stbl | コンテナ | 子Box群 |
| ftyp, mvhd, tkhd, mdhd | リーフ | 固定レイアウトのフィールド |
| stts, stsz, stco, stss, ctts | リーフ | テーブルデータ |
| mdat | リーフ | 映像・音声の生データ |

```
例: ftypボックス（20バイト）のバイナリ

00000000: 00 00 00 14  66 74 79 70  69 73 6F 6D  00 00 00 00
          ^^^^^^^^^^^^  ^^^^^^^^^^^^  ^^^^^^^^^^^^
          size=20       type="ftyp"   brand="isom"

00000010: 69 73 6F 6D
          ^^^^^^^^^^^^
          compatible="isom"
```

### sizeフィールドの特殊値

4バイト = 32bit = 最大約4GB。mdatが4GBを超える場合は拡張サイズが使われる。

```
size > 1 → そのままBoxサイズ（通常）
size = 1 → 直後の8バイトが64bitサイズ（4GB超のBox用）
size = 0 → ファイル末尾まで続く（最後のBoxのみ）
```

```
通常のBox（size ≤ 4GB）:
┌──────────────┬──────────────┬─────────────────┐
│  size (4B)   │  type (4B)   │  data           │
└──────────────┴──────────────┴─────────────────┘

拡張サイズのBox（size > 4GB）:
┌──────────────┬──────────────┬──────────────────────────┬─────────────────┐
│  size (4B)   │  type (4B)   │  largesize (8B)          │  data           │
│  = 1         │              │  実際のサイズ（64bit）     │                 │
└──────────────┴──────────────┴──────────────────────────┴─────────────────┘
```

moovやftypなど他のBoxは通常4GBを超えないため、拡張サイズは実質**mdatでしか使われない**。

## 2. トップレベル構造

通常のMP4ファイルは以下の3つのトップレベルBoxで構成される:

```
MP4ファイル全体:
┌─────────────────────────────────┐
│  ftyp  │  ファイルタイプ宣言     │  数十バイト
├─────────────────────────────────┤
│  moov  │  メタデータ・インデックス │  数KB〜数MB
├─────────────────────────────────┤
│  mdat  │  実際の映像・音声データ  │  ファイルの大部分
└─────────────────────────────────┘
```

### 2-1. ftyp（File Type Box）

ファイル形式のバージョン宣言。再生ソフトが最初に読んで「これはMP4だ」と判断する。

```
ftyp の中身:
  major_brand       = "isom"       ← ISO Base Media 準拠
  minor_version     = 0
  compatible_brands = ["isom", "mp41", "avc1"]  ← 互換性のあるプロファイル
```

brandの値によってMP4のバリエーション（通常のMP4、DASH用fMP4、3GPなど）が区別される。

### 2-2. moov（Movie Box）— 最重要

**ファイル全体のメタデータとインデックス**を格納する。再生に必須で、これがないと再生不能。

### 2-3. mdat（Media Data Box）

**実際の圧縮済み映像・音声データ**がそのまま並んでいる。mdat単体にはインデックス情報がないため、moovなしでは「どのバイトがどのフレームか」がわからない。

## 3. moov の内部構造

moovはMP4の心臓部。以下のような入れ子構造になっている:

```
moov
├─ mvhd          ← Movie Header: 全体の時間情報
├─ trak          ← Track: 映像トラック
│   ├─ tkhd      ← Track Header: トラック基本情報
│   └─ mdia      ← Media: メディア詳細
│       ├─ mdhd  ← Media Header: 時間スケール
│       ├─ hdlr  ← Handler: メディアタイプ（video/audio）
│       └─ minf  ← Media Information
│           ├─ vmhd/smhd  ← Video/Sound Media Header
│           └─ stbl       ← Sample Table ★ ここが核心
│               ├─ stsd   ← Sample Description（コーデック情報）
│               ├─ stts   ← Time-to-Sample（各サンプルの時間長）
│               ├─ stsc   ← Sample-to-Chunk（サンプルのチャンク割当）
│               ├─ stsz   ← Sample Size（各サンプルのバイト数）
│               ├─ stco   ← Chunk Offset（各チャンクのファイル内位置）
│               ├─ stss   ← Sync Sample（キーフレーム一覧）
│               └─ ctts   ← Composition Time Offset（表示順補正）
└─ trak          ← Track: 音声トラック
    └─ (同様の構造)
```

### 3-1. mvhd（Movie Header Box）

ファイル全体の時間情報を保持する。

```
mvhd の主なフィールド:
  timescale    = 1000      ← 時間の基本単位（1秒 = 1000 tick）
  duration     = 120000    ← 総再生時間（120000 / 1000 = 120秒）
  rate         = 1.0       ← 再生速度
  next_track_ID = 3        ← 次に追加するトラックのID
```

### 3-2. trak / tkhd（Track / Track Header）

各トラック（映像、音声、字幕）が1つずつ trak Box になる。

```
tkhd の主なフィールド:
  track_ID     = 1
  duration     = 120000
  width        = 1920.0    ← 映像トラックのみ
  height       = 1080.0    ← 映像トラックのみ
```

### 3-3. stbl（Sample Table Box）— 核心部分

**「mdatのどのバイト位置にどのフレームがあるか」を記録するインデックステーブル群**。シーク・再生のすべてがこの情報に依存する。

#### stsd（Sample Description）— コーデック情報

トラックで使われているコーデックの詳細。デコーダの初期化に必要。

```
映像トラックの stsd の例:
  format   = "avc1"          ← H.264
  width    = 1920
  height   = 1080
  avcC     = {               ← H.264固有のデコーダ設定
    profile  = High
    level    = 4.0
    SPS/PPS  = (バイナリ)     ← デコーダ初期化パラメータ
  }
```

#### stts（Time-to-Sample）— フレームの時間長

各サンプル（フレーム）の再生時間をランレングス圧縮で記録。

```
stts の例（30fps映像、timescale=90000）:
  entry_count = 1
  entries = [
    { sample_count = 3600, sample_delta = 3000 }
    ← 3600フレームがすべて3000 tick長（= 3000/90000 = 1/30秒）
  ]
```

可変フレームレートの場合はentryが複数になる。

#### stsz（Sample Size）— 各フレームのバイトサイズ

すべてのサンプルのサイズを個別に記録。

```
stsz の例:
  sample_count = 3600
  entry_size = [
    42301,   ← フレーム1: 42,301バイト（Iフレーム、大きい）
    5204,    ← フレーム2: 5,204バイト（Bフレーム、小さい）
    4891,    ← フレーム3: 4,891バイト（Bフレーム）
    12044,   ← フレーム4: 12,044バイト（Pフレーム、中くらい）
    ...
  ]
```

この情報と stco を組み合わせることで、任意のフレームのファイル内位置を計算できる。

#### stco / co64（Chunk Offset）— データのファイル内位置

mdat内のチャンク（サンプルをまとめた塊）の開始バイト位置を記録。

```
stco の例:
  entry_count = 900
  chunk_offset = [
    40,        ← チャンク1はファイルの40バイト目から
    85302,     ← チャンク2は85,302バイト目から
    170891,    ← チャンク3は170,891バイト目から
    ...
  ]
```

stco は32bitオフセット（最大4GB）、co64は64bitオフセット（4GB超のファイル用）。

#### stsc（Sample-to-Chunk）— サンプルとチャンクの対応

何番目のサンプルが何番目のチャンクに入っているかのマッピング。

```
stsc の例:
  entries = [
    { first_chunk=1,  samples_per_chunk=4, sample_description_index=1 }
    ← チャンク1以降、各チャンクに4サンプルずつ格納
  ]
```

#### stss（Sync Sample）— キーフレーム一覧

Iフレーム（ランダムアクセスポイント）のサンプル番号を列挙。**シークはこのフレームにしか飛べない。**

```
stss の例（GOP 30 の場合）:
  entries = [1, 31, 61, 91, 121, ...]
  ← 30フレームごとにキーフレーム
```

シークバーで2:35に飛びたい場合、プレーヤーは:
1. stss で 2:35 以前の最も近いキーフレームを探す
2. stco + stsc + stsz でそのフレームのファイル内位置を計算
3. そこからデコードを開始

#### ctts（Composition Time Offset）— 表示順補正

Bフレームがある場合、デコード順と表示順が異なるため、その差分を記録する。

```
GOP: I B B P の場合

デコード順:  I(0) P(3) B(1) B(2)  ← Pを先にデコードしないとBが復元できない
表示順:      I(0) B(1) B(2) P(3)  ← 視聴者が見る順序

ctts はこの「デコード順 → 表示順」の時間差を記録
```

## 4. シークの仕組み — stblを使った具体例

「1:00（60秒）の地点にシーク」したときのプレーヤーの動き:

```
1. stts を見て「60秒 = サンプル番号1800」と計算
   （30fps × 60秒 = 1800）

2. stss を見て「1800以前の最寄りキーフレーム = サンプル1771」と特定
   （entries = [..., 1741, 1771, 1801, ...]）

3. stsc を見て「サンプル1771 = チャンク443に所属」と計算

4. stco を見て「チャンク443 = ファイルの 156,789,012 バイト目」と取得

5. stsz を見て「チャンク内でのサンプル1771の正確な開始位置」を計算

6. ファイルの 156,789,012 バイト目あたりから読み出し開始

7. サンプル1771（Iフレーム）からデコードを開始し、
   サンプル1800まで追いついたら表示開始
```

この一連の処理が**moovを読むだけでできる**のがMP4の設計の肝。mdat全体をスキャンする必要がない。

## 5. faststart — moovの配置問題

### 通常のエンコード出力

エンコーダーは映像を先頭から順に処理するため、各フレームのサイズや位置は最後まで確定しない。そのため自然な出力順は:

```
通常の MP4:
┌──────┬─────────────────────────┬──────────┐
│ ftyp │         mdat            │   moov   │
│ 数十B│    （映像・音声データ）    │  数MB    │
└──────┴─────────────────────────┴──────────┘
                                  ↑ ファイル末尾
```

### ローカル再生では問題ない

ファイル全体がディスク上にあるので、末尾のmoovを即座に読める。

### ストリーミング再生で問題になる

HTTPでファイルを先頭からダウンロードする場合:

```
1. ftyp を読む → OK
2. mdat に到達 → moovがないのでデコードできない
3. mdat を全部スキップして moov まで到達 → やっとインデックスが判明
4. mdat の先頭に戻ってデコード開始

→ 巨大な mdat を全部ダウンロードするまで再生開始できない
```

HTTP Range Requestに対応しているサーバーなら末尾を先に取得できるが、追加のリクエストが必要で再生開始が遅れる。

### faststart の解決策

**moov を ftyp の直後（mdat の前）に移動する。**

```
faststart 適用後:
┌──────┬──────────┬─────────────────────────┐
│ ftyp │   moov   │         mdat            │
│ 数十B│  数MB    │    （映像・音声データ）    │
└──────┴──────────┴─────────────────────────┘
         ↑ 先頭近く
```

こうすると:

```
1. ftyp を読む → OK
2. moov を読む → インデックス確定、コーデック情報も取得
3. mdat の先頭からデコード開始 → 即座に再生可能
```

**ffmpegでの指定:**
```bash
ffmpeg -i input.mp4 -c copy -movflags +faststart output.mp4
```

**注意点:** moovを前に移動すると、stco（チャンクオフセット）の値がすべてmoovのサイズ分だけズレるので、全エントリを書き換える必要がある。faststartはこの書き換えを自動でやっている。

**MediaConvertとの関連:** MediaConvertはストリーミング向けの出力（HLS/DASH）ではfMP4を使うため faststart は不要。単一MP4ファイル出力の場合は内部でmoov配置を最適化している。

## 6. fMP4（Fragmented MP4）— ストリーミング向けの構造

通常のMP4は1つの巨大なmoov + mdatだが、fMP4は**小さなフラグメントの連続**で構成される。

### 通常MP4 vs fMP4

```
通常の MP4:
┌──────┬──────────┬──────────────────────────────────────┐
│ ftyp │   moov   │               mdat                   │
│      │(全体の   │  (全フレームが1つのmdatに)              │
│      │ インデックス)│                                      │
└──────┴──────────┴──────────────────────────────────────┘

fMP4:
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬────
│ ftyp │ moov │ moof │ mdat │ moof │ mdat │ moof │ ...
│      │(初期化│(frag1│(frag1│(frag2│(frag2│(frag3│
│      │ のみ) │index)│data) │index)│data) │index)│
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴────
              ├── fragment1 ──┤├── fragment2 ──┤
```

### 各Boxの役割

**moov（初期化セグメント）:**
- 通常MP4と違い、stbl内のサンプルテーブルは**空**
- コーデック情報（stsd）のみ格納 → デコーダの初期化に使う
- インデックスは各フラグメントのmoofが持つ

**moof（Movie Fragment Box）:**
- 対応するmdatフラグメントのインデックス
- 通常MP4のstblと同等の情報（サンプルサイズ、時間長、キーフレームフラグ等）を持つ

```
moof
├─ mfhd          ← Fragment Header（フラグメント番号）
└─ traf          ← Track Fragment
    ├─ tfhd      ← Track Fragment Header（デフォルト値）
    ├─ tfdt      ← Track Fragment Decode Time（開始時刻）
    └─ trun      ← Track Run（各サンプルの詳細）
        entries = [
          { duration=3000, size=42301, flags=SYNC },  ← Iフレーム
          { duration=3000, size=5204,  flags=0 },     ← Bフレーム
          { duration=3000, size=4891,  flags=0 },
          ...
        ]
```

### fMP4のメリット

**1. ライブ配信に対応できる**

通常MP4はmoovに全フレームの情報が必要 → ファイルが完成するまでmoovを書けない → ライブは不可能。fMP4はフラグメントごとに完結するので、生成しながら配信できる。

**2. ABR（Adaptive Bitrate）に最適**

HLS/DASHでは品質ごとに別のセグメントを用意する。fMP4のフラグメント = セグメントなので、自然に対応する。

```
HLS配信の構造:
  init.mp4        ← moov（初期化セグメント、共通）
  segment_001.m4s ← moof + mdat（0〜6秒）
  segment_002.m4s ← moof + mdat（6〜12秒）
  segment_003.m4s ← moof + mdat（12〜18秒）
  ...
```

**3. 障害耐性**

通常MP4はmoovが壊れると全体が再生不能。fMP4は壊れたフラグメントだけスキップすればよい。

### CMAF（Common Media Application Format）

HLSとDASHで同じfMP4セグメントを共有するための規格。fMP4の制約付きサブセット。

```
従来: HLS用(.ts) と DASH用(.m4s) を別々にエンコード → ストレージ2倍
CMAF: 同じ fMP4 セグメントを HLS/DASH 両方で使う → ストレージ半分
```

## 7. まとめ: MP4構造の全体像

| 概念 | 通常MP4 | fMP4 |
|------|---------|------|
| インデックス | moov内のstblに全情報 | 各フラグメントのmoofに分散 |
| ファイル完成前の再生 | 不可（moovが必要） | 可能（フラグメント単位） |
| シーク | stss + stco で計算 | mfra（あれば）またはマニフェスト参照 |
| faststart問題 | あり（moovの位置に依存） | なし（フラグメントごとに完結） |
| 主な用途 | ダウンロード配信、ローカル保存 | HLS, DASH, ライブ配信 |

**MediaConvertとの関連:**
- 「File Group」出力 → 通常MP4
- 「HLS Group」「DASH ISO Group」「CMAF Group」出力 → fMP4（または MPEG-TS）
- 出力グループの選択は、このコンテナ構造の違いを選んでいることに等しい
