# Q&A: MP4 コンテナのバイナリ構造

日付: 2026-03-26

`02_01_mp4_structure.md` の補足Q&A。

## Q1: sizeが4バイトで表現できないサイズだったらどうするの？

`size` フィールドに `1` をセットすると、直後の8バイトが**64bit拡張サイズ（largesize）**として解釈される。

```
通常のBox（size ≤ 4GB）:
┌──────────────┬──────────────┬─────────────────┐
│  size (4B)   │  type (4B)   │  data           │
└──────────────┴──────────────┴─────────────────┘

拡張サイズのBox（size > 4GB）:
┌──────────────┬──────────────┬──────────────────────────┬─────────────────┐
│  size (4B)   │  type (4B)   │  largesize (8B)          │  data           │
│  = 1         │              │  実際のサイズ（64bit）     │                 │
└──────────────┴──────────────┴──────────────────────────┴─────────────────┘
```

4バイト = 32bit = 最大約4GB なので、mdatが4GBを超える場合にこれが使われる。moovやftypは通常4GBを超えないため、実質**mdatでしか使われない**。

`size = 0` は「このBoxはファイル末尾まで続く」という意味で、最後のBox（通常はmdat）で使われることがある。

## Q2: ftypの中のmajor_brandなどもBoxの入れ子になっているのか？

入れ子ではない。ftypの中身はBoxではなく、**固定レイアウトのバイナリフィールド**が並んでいるだけ。

MP4のBoxには2種類ある:

- **コンテナBox** — 中身が子Boxの集まり（moov, trak, stbl など）
- **リーフBox** — 中身が固定レイアウトのデータ（ftyp, mvhd, stts, stsz など）

```
コンテナBox（moovの例）:
moov
├─ mvhd    ← 子Box
├─ trak    ← 子Box
│   ├─ tkhd  ← 孫Box
│   └─ mdia  ← 孫Box
└─ trak

リーフBox（ftypの例）:
ftyp
┌──────────┬──────────┬────────────────┬───────────────┬─────────────────────┐
│ size(4B) │type(4B)  │major_brand(4B) │minor_ver(4B)  │compatible_brands(可変)│
└──────────┴──────────┴────────────────┴───────────────┴─────────────────────┘
 ← Boxヘッダ →← 子Boxではなく、ただのバイト列が直接並ぶ →
```

## Q3: コンテナBoxのsizeは子・孫を全部含んだサイズになるのか？

その通り。コンテナBoxの `size` は**自身のヘッダ(8B) + 子Box全体のバイト数の合計**。

```
moov (size = 8 + 子Box全部)
├─ mvhd (size = 116)
├─ trak (size = 8 + 孫Box全部)
│   ├─ tkhd (size = 92)
│   └─ mdia (size = 8 + ...)
└─ trak (size = 8 + 孫Box全部)

moov.size = 8(moovヘッダ) + mvhd.size + trak1.size + trak2.size
```

パーサーはmoovのsizeで範囲を把握し、その中で子Boxのヘッダ（size + type）を順番に読んで再帰的にツリーを辿る。sizeがあるおかげで、**知らないBoxでも中身を解釈せずスキップできる**。

## Q4: trakはオフセット情報（自分がmoov内のどこにあるか）を持たないのか？

持たない。trakはmoovの中に順番に並んでいるだけで、パーサーはmoovの範囲内を先頭から順にBoxヘッダを読んでいくのでオフセットなしで辿り着ける。

```
moov (size = 1200) の中を先頭から読む:
  ┌─ mvhd (size=116) → type="mvhd" → 116バイト進める
  ├─ trak (size=540)  → type="trak" → 映像トラック
  └─ trak (size=536)  → type="trak" → 音声トラック
                      → moovの残りが0 → 終了
```

ただし、ファイル全体の中で特定のBoxを高速に見つけるためのグローバルな索引はfMP4の `mfra`（Movie Fragment Random Access）やMKVの `SeekHead` にはあるが、通常のMP4にはない。トップレベルBoxは先頭から順にスキャンする設計。

## Q5: YouTubeの動画をMP4でダウンロードするツールは何をしているのか？

YouTubeはDASH（Adaptive Bitrate Streaming）で配信しており、サーバーには完成したMP4ファイルが置いてあるわけではない。

```
YouTubeサーバー上の構成（1本の動画に対して）:
  映像: 144p, 360p, 720p, 1080p, 4K ...（各解像度ごとに別ストリーム）
  音声: 低音質, 中音質, 高音質 ...（各品質ごとに別ストリーム）
  ※ 映像と音声が分離されている
```

ダウンロードツールがやっていること:

```
1. WebページのHTML/JavaScriptを解析して各品質のストリームURLを取得

2. 指定品質の映像ストリームと音声ストリームを全セグメントダウンロード

3. ffmpeg等で映像と音声をMP4コンテナにMux（多重化）
   → moov, mdat を構築して1つのMP4ファイルとして出力
```

```
YouTube側:
  映像(1080p): [seg1][seg2][seg3]...  ← WebM or fMP4 (VP9/AV1/H.264)
  音声(高音質): [seg1][seg2][seg3]...  ← WebM or fMP4 (Opus/AAC)

ダウンロードツール:
  映像全セグメント取得 ─┐
                      ├→ ffmpeg -i video -i audio -c copy output.mp4
  音声全セグメント取得 ─┘
```

コーデックがMP4互換（H.264 + AAC）なら `-c copy`（再エンコードなし）でコンテナだけ組み替えるので一瞬で終わる。VP9 + Opus の場合はMP4に入れられないのでWebMのまま保存するか、再エンコードが必要。

ストリーミングを傍受してリアルタイムに組み立てているわけではなく、セグメントのURLがわかっているので普通のHTTPダウンロードで全部取ってきて最後にまとめている。

## Q6: トラックごとにコーデック情報を持っているのか？mvhdに持たせればよいのでは？

各トラックのコーデックは異なるため、**トラックごとに持つ必要がある**。

```
trak 1（映像）の stsd → "avc1" (H.264), profile=High, level=4.0, SPS/PPS=...
trak 2（音声）の stsd → "mp4a" (AAC), sample_rate=48000, channels=2
trak 3（字幕）の stsd → "tx3g" (テキスト字幕)
```

mvhdが持っているのは**ファイル全体で共通の情報**（総再生時間、タイムスケール等）だけ。コーデック情報をmvhdに混在させると、映像が2トラック・音声が3言語といったケースで破綻する。

また、トラックは細切れのセグメントではなく、**1本のストリーム全体**を指す。

```
trak 1 = 映像トラック（0:00〜2:00の映像全部）
trak 2 = 音声トラック（0:00〜2:00の音声全部）
trak 3 = 字幕トラック（0:00〜2:00の字幕全部）
```

mdat内では映像と音声がインターリーブされてバラバラに配置されているが、各トラックのstbl（サンプルテーブル）が「mdatのどのバイトがこのトラックのどのサンプルか」を管理しているので、論理的には1本にまとまる。

## Q7: 映像と音声のインターリーブ（muxing）はMP4のどこで制御されているのか？

MP4の仕様上インターリーブを制御する専用のBoxがあるわけではない。**エンコーダー/マクサー（ffmpegなど）がmdatにデータを書き込む時点で配置を決め、stcoがその結果を記録している**。

```
mdat の中身（物理的なバイト配列）:
  [映像0.0-0.5s][音声0.0-0.5s][映像0.5-1.0s][音声0.5-1.0s]...

各トラックのstcoがどのバイトが自分のデータかを管理:

trak 1（映像）の stco:     trak 2（音声）の stco:
  chunk 1 → offset 40       chunk 1 → offset 50000
  chunk 2 → offset 100000   chunk 2 → offset 150000
  chunk 3 → offset 200000   chunk 3 → offset 250000
```

書き込み時（エンコーダー）:
- 映像チャンクと音声チャンクを交互にmdatに書き込み、書き込んだ位置をstcoに記録

読み込み時（プレーヤー）:
- 各トラックのstcoを読んで「映像の次のチャンクはoffset 100000、音声の次はoffset 150000」と把握して読みに行く

インターリーブの粒度（何秒分ごとに交互にするか）もエンコーダーが決める:

```bash
# ffmpegでインターリーブ間隔を指定する例
ffmpeg -i input.mp4 -c copy -movflags +faststart \
  -interleave_delta 500000 output.mp4
#                   ^^^^^^ マイクロ秒単位（500000 = 0.5秒）
```

間隔が短いほどシーク後の再生開始が速い（映像と音声の両方がすぐ揃う）が、細かくしすぎるとオーバーヘッドが増える。

## Q8: stszには全フレームのサイズが1つずつ記録されているのか？

その通り。30fps × 60秒 = 1800フレームなら1800エントリ。

```
stsz の構造:
  sample_size  = 0         ← 0 = 各サンプルのサイズが異なる（個別記録）
  sample_count = 1800
  entry_size = [
    42301,    ← フレーム1（Iフレーム）
    5204,     ← フレーム2（Bフレーム）
    4891,     ← フレーム3（Bフレーム）
    12044,    ← フレーム4（Pフレーム）
    ...
    3298,     ← フレーム1800
  ]

データ量: 4バイト × 1800 = 7,200バイト（約7KB）
```

`sample_size` が0以外の場合は「全サンプルが同じサイズ」という意味で、エントリの列挙が不要になる。非圧縮音声（PCM）のように全サンプルが同一サイズのトラックで使われる。映像トラックではI/P/Bフレームでサイズが大きく異なるため、実質必ず個別記録（`sample_size = 0`）になる。

2時間の映画（30fps）だと 30 × 7200 = 216,000エントリで約864KB。moovが数MBになる主な原因の一つがこのstszとstco。

## Q9: stszのサイズ情報から特定のフレームの位置を計算するのか？

stszだけではフレームの位置はわからない。stszは「サイズ」、stcoは「チャンクのファイル内位置」、stscは「サンプルとチャンクの対応」で、**3つを組み合わせて初めてフレームの位置が特定できる**。

```
知りたいこと                       → 使うテーブル
──────────────────────────────────────────────────
フレームNは何バイト？                → stsz
フレームNはどのチャンクに属する？      → stsc
そのチャンクはファイルの何バイト目？    → stco
チャンク内でのフレームNの正確な位置は？ → stsz（先頭からの累積）
```

具体例:

```
フレーム10 を読みたい場合:

1. stsc を見る → 「チャンク1本あたり4サンプル」
   → フレーム10 はチャンク3の2番目

2. stco を見る → 「チャンク3 はファイルの 200,000 バイト目から」

3. stsz を見る → チャンク3 の1番目（フレーム9）のサイズは 5,100 バイト
   → フレーム10 の開始位置 = 200,000 + 5,100 = 205,100 バイト目

4. stsz を見る → フレーム10 のサイズは 4,800 バイト
   → 205,100 から 4,800 バイト読めばフレーム10 が得られる
```

3つに分かれている理由は**データの圧縮効率**:

- stco: チャンク数分のエントリで済む（サンプル数より少ない）
- stsc: ランレングス圧縮（「チャンク1〜900は全部4サンプルずつ」→ 1エントリ）
- stsz: これだけは全サンプル分必要（サイズはバラバラなので圧縮できない）

全サンプルのオフセットを個別に持つより、この3つに分けたほうがmoov全体のサイズが小さくなる。
