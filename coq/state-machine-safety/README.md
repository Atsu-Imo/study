# 状態機械の安全性証明 — 「危険な状態に到達しない」

> ⚠️ これはカリキュラム 09〜11 (ミニ言語 / プログラム等価性 / ホーア論理) 相当の
> **先取りプレビュー**です。「Rocq で実務的な検証は何ができるのか」の雰囲気を掴むための例で、
> 使っている道具（`Inductive` な命題、命題への帰納法）はトピック 06 以降で本格的に学びます。
> いま全部わからなくても問題ありません。「証明の形」を眺めてください。

## 1. 概要

注文(order)の状態機械をモデル化し、
**「支払っていないのに発送済み(`shipped && !paid`)」という危険状態には決して到達しない**
ことを証明します。

これはソフトウェア検証の最も実務的な使い方のひとつ
「**到達可能性の安全性 (safety: bad state unreachable)**」の最小例です。
セキュリティの文脈（権限昇格しない、二重決済しない）や、
プロトコル検証（合意が壊れない）と同じ骨格を持ちます。

## 2. 背景知識（経験者向けの対比）

| 普通のソフトウェア開発 | Rocq でのモデル化 |
|---|---|
| `struct State { ... }` | `Record State := mkState { ... }` |
| 状態遷移関数 / if 文のガード | `Inductive step : State -> State -> Prop`（許可された遷移を列挙） |
| ユニットテスト（個別ケース） | `Theorem`（全到達状態を一度に保証） |
| 「不正な状態にならないはず」 | `invariant` を定義し、帰納法で**証明** |

ポイントは **「許可された遷移だけを `Inductive` で列挙する」** こと。
そこに書かれていない遷移（例：`Pending` から直接 `Shipped`）は
**型レベルで存在しない**ので、考慮対象から消えます。
これは Go で「あらゆる入力を if でチェックする」のとは逆の発想です。

## 3. コード解説（証明の骨格）

安全性は **帰納法による不変条件 (invariant) の証明** という定石で示します。

```coq
(* 守りたい性質：発送済みなら必ず支払い済み（= bad の否定） *)
Definition invariant (s : State) : Prop :=
  shipped s = true -> paid s = true.
```

証明は4段階：

1. **`init_inv`** — 初期状態が不変条件を満たす（base case）。
   初期は未発送なので前提 `false = true` が矛盾し、自明に成立。
2. **`step_preserves_inv`** — どの1ステップ遷移も不変条件を保つ（preservation）。
   `inversion Hstep` で遷移を3ケースに分解し、各遷移で確認する。
   **肝は「発送 `step_ship`」のケース**：遷移のガード `paid s = true` が前提にあるので、
   発送後も `paid = true` が保たれる。**「ガードが不変条件を守る」**という構図がそのまま証明になる。
3. **`reachable_inv`** — `reachable` への帰納法で (1)(2) を繋ぎ、
   **到達可能な全状態**が不変条件を満たすと結論する。
4. **`safety`** — ゆえに `bad`（未払いで発送済み）には到達不能。

```coq
Theorem safety : forall s, reachable s -> ~ bad s.
```

> 「テストは存在の証明、証明は不在の証明」。
> ここで示したのは **「危険状態が存在しない」** という、テストでは原理的に到達できない保証です。

末尾の `normal_flow_reachable` は逆に
「初期 → 支払い → 発送」という**正常系が確かに到達可能**なことを示す例です
（モデルが厳しすぎて何もできない、という事故を防ぐサニティチェック）。

## 4. 演習

### 基礎
`main.v` を RocqIDE で開き、`safety` の証明を1ステップずつ実行して、
各タクティクでゴール・前提がどう変化するか観察する。
特に `step_preserves_inv` の3つのサブゴール（pay / ship / cancel）の違いを見る。

### 応用
別の不変条件
**「キャンセル済みなら発送されていない (`cancelled s = true -> shipped s = false`)」**
を `invariant2` として定義し、`init` で成立・各 `step` で保存されることを証明せよ。
（ヒント：`step_ship` のガード `cancelled s = false` が効く）

### チャレンジ
状態に「返金済み `refunded : bool`」フラグを追加し、遷移
`step_refund`（支払い済みのときだけ返金できる）を加えたうえで、
新たな危険状態 **「支払っていないのに返金済み」** に到達しないことを証明せよ。
既存の証明がどこまで再利用でき、どこを直す必要があるか確認する。

## 5. まとめ

- `Inductive step` で **「許可された遷移だけ」** を列挙すると、不正遷移は型レベルで消える。
- 安全性 = **「危険状態に到達しない」** は、`invariant` を立てて
  **(init で成立) + (step で保存) → (全到達状態で成立)** という帰納法の定石で示す。
- 遷移の **ガード（前提条件）が不変条件を守る**、という対応が証明の中心。
- これはセキュリティ・プロトコル・業務ロジックの「あってはならない状態」検証に直結する、
  Rocq の最も実務的な使い方のひとつ。
