## 🔬 深掘りメモ

### CVE-2026-0257 — PAN-OS GlobalProtect 認証バイパス（結論: 自社は GlobalProtect 未使用のため非該当）
- **根本原因**: 認証オーバーライドCookieに暗号的な完全性チェックが無い（CWE-565）。Cookie暗号化用の証明書を**他機能と共用**していると、攻撃者がCookieを偽造でき、認証チェックを飛ばして「認証済みVPNユーザー」になりすませる。
- **悪用3条件（AND）**: ①GlobalProtect ポータル/ゲートウェイ有効 ②認証オーバーライドCookie有効 ③証明書が専用でなく共用。Cloud NGFW は影響なし。
- **深刻度**: CVSS 7.8 High（`AV:N/AC:L/PR:N/UI:N`、無認証・低難易度）。当初 Medium → 悪用確認で High 格上げ。`SC:H/SI:H`＝内部ネットワーク侵入の質的怖さ。
- **実観測 (Rapid7)**: 公式公開 5/13 → 悪用開始 5/17。admin狙い、偽MAC `aa:bb:cc:dd:ee:ff`、マシン名 `GP-CLIENT`/`DESKTOP-GP01`、`POST /ssl-vpn/{hipreport,getconfig}.esp`、送信元 Vultr/Dromatics。複数顧客で悪用成功。
- **対応**: パッチ最優先（12.1≥12.1.4-h6、11.2≥11.2.12、11.1≥11.1.15、10.2は各hotfix）。緩和は機能無効化＋証明書の専用分離。

### CVE-2026-42189 / CVE-2026-46673 — russh untrusted-length OOM-DoS（学習トピック）
**脆弱コード** (`russh/src/server/encrypted.rs` の `read_userauth_info_response`):
```rust
let n = map_err!(u32::decode(r))?;          // クライアント自己申告の個数（検証なし）
let mut responses = Vec::with_capacity(n);  // その個数で即メモリ確保 → n=0x10000000 で ~6.4GB
```
~50バイトのパケットで6.4GB確保を誘発 → OOM。pre-auth・無認証・1パケットで成立（CVSS High 7.5）。

**本物の修正（自己申告ではなく物理的裏付けでクランプ）**:
```rust
let max_responses = r.remaining_len().saturating_add(3) / 4; // 残りバイトが含みうる最大数
let n = (n as usize).min(max_responses);                     // 嘘の n を無害化
let mut responses = Vec::with_capacity(n);                   // クランプ後なら事前確保OK
```

**なぜ事前確保するのか / 動的pushではダメか**（要点）:
- 事前確保 `with_capacity(n)` は純粋な**性能最適化**（realloc＋コピーを回避）。機能上は不要。
- **「読みながら push」が最も安全**。メモリ消費が「攻撃者が実際に転送したバイト数」に比例するため、増幅攻撃（50B→6.4GB ≈ 1億倍）が構造的に消える。push 方式の増幅率は約1倍。
- 二択ではなく折衷が定石: **事前確保のサイズは "宣言された量(n)" ではなく "確認できる量(残バイト/フレーム上限)" で決める**。
- 一般化（言語非依存・CWE-770）: Go の `make([]T, 0, n)` も全く同じ罠。ワイヤ上の length フィールドを、その裏付けを確認せず確保サイズに使うな。
- 防御の定石: ①入力の物理的裏付けで上限化（最強）②プロトコル仕様の絶対上限でreject ③遅延確保/incremental push ④ストリーム全体にフレーム長上限（=CVE-2026-46673 の 256KiB 修正）。
- 実務デフォルト: ホットパス以外は素直に push（安全側）。事前確保するならクランプ必須。
