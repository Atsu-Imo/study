### CVE-2022-0492 — Linux Kernel cgroups v1 権限昇格 / コンテナ脱出（KEV 入り 2026-06-02、FCEB 期限 6/5）

**一行結論**: cgroup v1 の `release_agent` 書き込みに「適切な名前空間での `CAP_SYS_ADMIN` チェック」が無かった古い穴（修正から4年）。**デフォルト堅牢化（Seccomp/AppArmor/SELinux）が効いていれば脱出はほぼ防げる**が、それらを外した特権コンテナ構成は今まさに狙われている。

**根本原因（CWE-862: 認可の欠如）**
- 脆弱関数: `kernel/cgroup/cgroup-v1.c` の `cgroup_release_agent_write()`。
- バグの本質: 「`release_agent` ファイルへ書く側が管理者権限を持つか」を**そもそも検証していなかった**。
- 修正パッチ（要点）:
  ```c
  if ((of->file->f_cred->user_ns != &init_user_ns) ||
      !capable(CAP_SYS_ADMIN))
      return -EPERM;
  ```
  → **初期ユーザー名前空間 (init_user_ns) における** `CAP_SYS_ADMIN` を要求。ここが肝で、ユーザー名前空間を `unshare` して「その中での疑似 root」を持っていても弾かれるようになった。

**`release_agent` とは / なぜ致命的か**
- cgroup 内の最後のプロセスが終了した時に、管理者が指定したプログラムを実行する仕組み。
- その実行は **「初期名前空間で全 capability を持つ root プロセス」** として走る。＝攻撃者が実行バイナリのパスを書ければ、ホスト上で root 任意コード実行になる。

**攻撃チェーン（step by step）**
1. `unshare()` で新しい user namespace + cgroup namespace を作る（非特権ユーザーでも可）。
2. その名前空間内で書き込み可能な cgroupfs を mount。
3. root cgroup の `release_agent` ファイルにアクセス。
4. 攻撃者バイナリのパスを `release_agent` に書き込む。
5. cgroup 内のプロセスを終了させてエージェントを発火 → ホスト root で実行。

**悪用成立の前提（5条件すべて必要）**
1. コンテナが root 実行、または `no_new_privs` 無効。
2. AppArmor も SELinux も無効。
3. Seccomp 制限なし（`unshare`/`mount` を塞いでいない）。
4. ホストで非特権 user namespaces が有効。
5. コンテナが cgroup v1 ルートに属する（**v2 のみなら非該当**）。

**自社環境のチェック観点**
- 使っているのは cgroup v1 か v2 か（最近のディストリ/EKS は v2 既定が多く、その場合は構造的に非該当）。
- Pod/コンテナの Seccomp プロファイル（`RuntimeDefault` 以上か）、AppArmor/SELinux 有効か。
- 特権コンテナ・`CAP_SYS_ADMIN` 付与・hostPath で cgroupfs を渡している箇所が無いか。
- 確認用: Unit42 の診断イメージ `us-central1-docker.pkg.dev/twistlock-secresearch/public/can-ctr-escape-cve-2022-0492`。

**なぜ「今」KEV 入りか（実務的含意）**
- 脆弱性自体は2022年だが、2026-06-02 に「悪用が観測された」として KEV 追加・連邦機関は 6/5 までに対応。＝古い穴でも**パッチ未適用 or 堅牢化を外した構成**が現役で狙われている。
- 対応の本筋: ①カーネルを修正済みに（4年前の修正なので最新化で解消）②防御の多層化（Seccomp+LSM を外さない）③可能なら cgroup v2 へ。単独 CVE というより「コンテナ堅牢化を省くと刺さる」典型例。

### CVE-2026-47429（ほか） — Vitest UI/API サーバの任意ファイル読み取り→RCE（公開 2026-06-01）

**一行結論**: Vitest の UI/API サーバを **localhost 以外に公開している（`api.host`）か、Windows で UI/Browser Mode を動かしている**場合に、無認証でファイル読み取り→RCE まで至る。**localhost 既定のローカル/CI 実行なら基本セーフ**。

**何が起きるか**
- 中心: `CVE-2026-47429`（`GHSA-5xrq-8626-4rwp`、CVSS 9.8 / `AV:N/AC:L/PR:N/UI:N`）。
- 入口: `/__vitest_attachment__` API が `isFileServingAllowed` を誤用し、ファイル操作前に `cleanUrl`（URL正規化）を通さない → パストラバーサル検証をすり抜け、プロジェクト外の任意ファイルを読める。
- RCE 昇格: `saveTestFile`（任意スクリプトをテストファイルとして書き込み）＋ `rerun`（再実行）の組み合わせでコード実行。
- 関連クラスタ（同時期公開）: `CVE-2026-47428`（反射型スクリプトインジェクション、クエリパラメータ未サニタイズ、クロスプラットフォーム）、Chrome DevTools Protocol 露出によるプロジェクトroot への悪性ファイル投下（クロスプラットフォーム）。

**影響範囲（成立条件）— ここが実務上の肝**
- 危険なのは「**そのUI/APIポートにネットワーク到達できる経路がある**」とき。無認証なので、正規メンバーに限らず到達できる第三者が攻撃可能。
- 該当パターン: ① `--api.host` / `api.host` で localhost 以外にバインドして公開 ② Windows 上で UI / Browser Mode 実行（パストラバーサルは Windows 固有の `\\?\\..\\` 挙動を突く。Linux は非該当）。
- 非該当に近い: localhost 既定のローカル開発 / CI 実行（公開サーバではない）。
- ＝「複数人で共有しているか」ではなく「外から叩ける口が空いているか」が判定軸。

**対応**
- アップグレードが本筋。情報源で版に差（公式GHSAは `>=4.1.0`、複数報道は `4.1.6` / `5.0.0-beta.3`）→ 安全側で **最新の `4.1.6+`（または 5.0 系）** に。
- 4.1.0 以降は `allowWrite` / `allowExec` フラグが追加され、API が非 localhost バインド時は**両方デフォルト無効**（UIは読み取り専用）。
- すぐ上げられない場合の緩和: `api.host` で外部公開しない（localhost のまま）だけで主経路は塞がる。
