### 📝 本日深掘りした内容（セッションログ）

#### 1. Aurora PostgreSQL 18 対応の「嬉しいこと」
- **B-tree skip scan**: 複合インデックスの先頭列を `WHERE` で指定しなくても、先頭列の値の種類が少なければプランナが内部で各値を回して活用してくれる → 冗長な複合インデックスを減らせる（先頭列が高カーディナリティだと効果薄）。
- **非同期 I/O**: シーケンシャル/ビットマップスキャンの読み取りが向上。ストレージ分離アーキの Aurora と特に好相性、コード変更不要。
- **pg_roaringbitmap**: 巨大 ID 集合の集合演算（AND/OR/差分）を高速化。会員フラグ・セグメント集計・視聴者集合などに。
- **UUIDv7 ネイティブ**: 時刻順ソート可能で挿入性能・インデックス効率が良い（v4 の断片化を回避）。
- 注意: Aurora のメジャーVUは自分で実施が必要。`EXPLAIN` で skip scan が効くか検証してから上げる。

#### 2. postgres-protocol 脆弱性クラスタ（RUSTSEC-2026-0178〜0180）の修正差分
- 影響範囲: tokio-postgres / postgres 系（**sqlx は独自実装で非該当**）。`cargo tree -i postgres-protocol` で確認。脅威方向は「サーバ→クライアント」。
- **① SCRAM 反復回数（0179, High）** [d40097a](https://github.com/rust-postgres/rust-postgres/commit/d40097a36a85068ea50a3afbf0ce154ba439e7f0): `MAX_ITERATION_COUNT: u32 = 100_000` を定義し、重い `hi()`(PBKDF2) を呼ぶ**前**に上限チェック。→ 「高コスト処理の手前で検証」。
- **② DataRow フィールド数不一致（0178, Medium）** [7a00ffa](https://github.com/rust-postgres/rust-postgres/commit/7a00ffa9ad4d951ec0a4564b52f1780fa9d353c1): `Row::new()`（構築の入口）で `ranges.len() != columns().len()` を検査しエラー化。→ 「不正な行はそもそも生成させない＝コンストラクタで不変条件確立」。
- **③ hstore デコード panic（0180, Medium）** [a7cf84b](https://github.com/rust-postgres/rust-postgres/commit/a7cf84b5c46431cbca9d8ff50508c23f446efa7d): panic する `split_at` → `Option` を返す `split_at_checked` + `?` でエラー伝播。→ 「panic 版/Result 版ペアでは信頼できない入力に `_checked` を選ぶ」。
- 教訓: **メモリ安全（panic で止まる）≠ 可用性（落とさない）**。信頼できない入力のデコードでは panic 経路を Result に移す。

#### 3. File Browser 脆弱性クラスタ（CVE-2026-54090〜54097, すべて v2.63.6 で修正）
- **54090 コマンド注入（8.7）**: 先頭トークンのみ許可リスト照合 → raw 文字列を `sh -c` に渡しシェルメタ文字で連鎖。default root で実質 RCE。教訓: 検証する形＝実行する形にする。`exec.Command(name, args...)` で引数配列にすればシェル解釈が挟まらない。
- **54091 認可バイパス（7.5, 未認証）**: 公開共有でルートを rebase したが deny ルールは元基準のまま → 基準系不一致で deny 回避・情報漏洩。
- **54092 ログイン DoS（8.7, 未認証）**: `/api/login` がパスワード長未制限 → ハッシュ計算で資源枯渇。②の SCRAM と同型「暗号処理前にサイズ未検証」。
- **54097 IDOR（7.2）**: 共有削除が前方一致のみで所有者未チェック → 他人の共有リンクを巻き添え削除。教訓: 操作対象に必ず所有者スコープ（`AND owner_id = ?`）を差す。
- まとめ: Go バックエンドの定番アンチパターン4種（コマンド注入 / 認可バイパス / 未認証DoS / IDOR）の教材セット。v1.x は修正なし＝v2 移行が必要。

#### 4. Jane Street「Formal methods and the future of programming」解説
- 主張: **AI エージェント時代になり形式手法のコスト/便益が逆転しつつある**。①証明コストを AI が下げる ②AI 生成コードの検証需要が増える ③形式的保証はエージェントへの強力なフィードバック。
- 象徴例: seL4 は「8,700 行の C を 25 人年で検証」＝従来は採算が合う領域が極小だった。
- 道具: 自社 OxCaml を主軸に Lean/Rocq/Agda/Iris を参考。形式手法を「高機能な型システムの延長」と捉える。
- 含意: 「証明の職人芸」より「**仕様を書く力・仮定を意識する力**」に学習を寄せると費用対効果が高い。
- **(a) seL4 が保証したもの**: 抽象仕様→実行仕様→C 実装のリファインメント連鎖で機能的正しさを証明。後にバイナリ⇔C 一致でコンパイラを信頼チェーンから除外、さらに integrity/confidentiality(非干渉)/authority confinement も証明。**保証しないもの＝仕様自体の正しさ・ハード・ブートアセンブリ・(当初)マルチコア/タイミング**。「verified＝仮定のもとでの条件付き主張」。※ seL4 は Coq でなく Isabelle/HOL。
- **(b) 証明支援系の使い分け**: Rocq=ソフト/PL 検証の本命（抽出・CompCert・Iris、学習中のこれ）／Lean4=数学形式化と勢い(mathlib)／Agda=依存型プログラミング・教育・HoTT／Isabelle/HOL=自動化重視の大規模検証(seL4)。
