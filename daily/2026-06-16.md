### 📚 今日の深掘りログ — P2Pネットワーク（Iroh起点）

**テーマ：** Iroh をきっかけに P2P ネットワークの仕組みを概念レベルで整理。

- **クライアント・サーバ vs P2P**
  - C/S は全員が中央サーバに繋ぎ、データは必ずサーバ経由。各ノードは住所を持たなくてよい。
  - P2P は全ノードが対等（peer）。データを当事者間で直接やりとり。例：BitTorrent / ビデオ通話 / IPFS。
  - メリット：低レイテンシ・サーバ帯域コスト減・スケール（参加者が増えるほど配り手も増える）・耐障害性/検閲耐性・プライバシー（サーバにコピーが残らない）。
  - デメリット：相手の発見が難しい・NAT/FWが壁・一貫性/認証が難しい。

- **アドレッシング："Dial Keys, not IPs"**
  - 個人デバイスは IP が変動し NAT 裏で固定住所を持てない。
  - そこで「住所(IP)」でなく「身元(公開鍵=NodeID)」で相手を指定。IP が変わっても鍵は不変＝追跡可能、かつ相手の本人性を暗号的に検証できる。
  - Iroh：鍵で指定 → 直接接続を試み → ダメなら relay にフォールバック。トランスポートは QUIC（UDPベース）でE2E暗号化。

- **NAT越え＝ホールパンチング**
  - NAT は「自分から出た通信の返事」しか内側に通さない → 外から直接繋げない。
  - 仕掛け：先に自分から外へ撃って NAT に穴(マッピング)を作り、相手のパケットを"返事"として通す。
  - 手順：①STUN で自分の外側IP:ポートを知る ②仲介サーバで相手の住所を交換 ③同時に撃ち合って両側の穴を揃える。以後サーバは退場し直接通信。
  - UDP が主戦場（TCPは順序が厳密で通しにくい）→ だから QUIC を使う。
  - **Symmetric NAT / CGNAT** は外側ポートが予測できず失敗しやすい → relay フォールバックが必須。

- **IP共有の話（IP:ポートで区別）**
  - IP共有は常態（家庭・モバイル・共有物件）。区別は IP単体でなく **IP:ポート**のペア＝「住所＋部屋番号」。
  - 普段の outbound 通信は各NAT層がポートで捌くので無問題。困るのは「外から繋がれる側」になるP2P/サーバ公開時。
  - 共有物件やモバイルの CGNAT（多段NAT）はポート開放不可・パンチング困難 → relay 行き。
  - 根本解決は **IPv6**（端末ごとに固有グローバルアドレス → NAT/共有が消え P2P が本来の姿に）。

- **"直接やりとり"の正体**
  - 流れるのは特別なデータではなく**アプリが運びたい実データ**（ファイルのチャンク・音声/映像フレーム・メッセージ等）。
  - プログラム視点では `conn.Write/Read` するバイト列で C/S と同じ。違いは `conn` の相手が"サーバ"か"相手端末"か**だけ**。
  - 2フェーズ構成：①確立フェーズ（公開鍵・アドレス候補・探り弾・暗号合意を仲介サーバ経由で交換）②データフェーズ（サーバを一切経由せず当事者間で実データ転送＝サーバにコピーが残らない）。

**次の深掘り候補：** STUN/TURN/ICE（WebRTCが標準化した発見・relay の枠組み、TURN=relay）/ DHT（仲介サーバすら無くす分散探索、BitTorrent・IPFS）/ Iroh か Go の最小UDPホールパンチを実コードで書く。
