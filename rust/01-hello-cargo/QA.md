# Q&A: 01-hello-cargo

日付: 2026-03-06

## Q1: editionが2021だけど古くない？最新はどのバージョン？

最新は **Edition 2024**。Rust 1.85.0（2025年2月リリース）で安定化された。

主な変遷:

| Edition | 安定化時期 |
|---------|-----------|
| 2015 | 初版 |
| 2018 | Rust 1.31 |
| 2021 | Rust 1.56 |
| 2024 | Rust 1.85 (2025-02) |

2024 editionの主な変更点:
- `gen` キーワードの予約（将来のジェネレータ用）
- `unsafe_op_in_unsafe_fn` がデフォルトで警告
- RPIT（impl Trait）のライフタイムキャプチャルールの変更
- `tail_expr_drop_order` の変更

2021でも学習には問題ないが、新規プロジェクトなら2024を使うのが良い。

## Q2: ダブルクォーテーション内で変数は自動展開される？シングルクォーテーションは使えない？データ型で使い分ける？

**ダブルクォーテーション `""`** は文字列（`&str` / `String`）。変数展開は文字列自体の機能ではなく、`println!` 等のフォーマットマクロの機能。普通の代入では展開されない。

```rust
let name = "Rust";
let s = "hello {name}";      // → そのまま "hello {name}"
println!("hello {name}");     // → "hello Rust"（マクロが展開）
```

**シングルクォーテーション `''`** は1文字（`char`型、Unicode）。文字列には使えない。

```rust
let c: char = 'R';        // OK
let s: &str = "Rust";     // OK
// let s = 'Rust';        // NG: コンパイルエラー
```

Goとの比較:
- Go: `""` = string, `''` = rune (int32), バッククォート = raw string
- Rust: `""` = &str, `''` = char (Unicode), `r#""#` = raw string

結論: **データ型で使い分け**（`""` = 文字列、`''` = char）。変数展開はフォーマットマクロの機能。

## Q3: `{:#?}` はどんなデータ型でも受け付けるのか？

すべてではなく、**`Debug` トレイトが実装されている型**のみ。

- 標準ライブラリの基本型（数値、文字列、bool、タプル、配列、Vec、HashMapなど）は `Debug` 実装済み
- 自作の構造体/enumは `#[derive(Debug)]` を付ければ使える。付けないとコンパイルエラー

`{:?}` と `{:#?}` の違い:
- `{:?}` → 1行表示: `Point { x: 1, y: 2 }`
- `{:#?}` → インデント付き整形表示（pretty-print）

Goとの比較:
- Go の `%v` はリフレクションベースでどんな型でも使える
- Rust の `{:?}` はコンパイル時に `Debug` トレイトの有無をチェック。明示的な実装が必要だが実行時コストが低い

## Q4: `#[derive(Debug)]` ってどういうこと？

**「この構造体に `Debug` の機能を自動生成してね」というコンパイラへの指示。**

```rust
#[derive(Debug)]    // コンパイラが Debug 用のコードを自動生成
struct Point {
    x: i32,
    y: i32,
}

// これで {:?} や {:#?} で表示できる
println!("{:?}", Point { x: 1, y: 2 });   // → Point { x: 1, y: 2 }
```

分解すると:
- `#[...]` → アトリビュート。型や関数にメタ情報を付ける仕組み（Goのstructタグに近い発想）
- `derive` → 指定したトレイト（≒Goのインターフェース）の実装コードを自動生成する
- `Debug` → 表示用トレイトの一つ

複数まとめて `#[derive(Debug, Clone, PartialEq)]` のように導出できる。

Goとの比較:
- Go は `%v` がリフレクションで何もしなくても動く
- Rust は明示的に宣言が必要だが、コンパイル時チェック＋実行時オーバーヘッドなし

詳細は `08-traits-and-generics` で学ぶ。今は「`#[derive(X)]` = Xの機能を自動実装してもらう仕組み」と覚えればOK。

## Q5: マクロって何？

**コンパイル前にコードを生成する仕組み。** 関数とは異なり、コンパイル時に展開されて実際のコードに置き換わる。

`println!` が関数ではなくマクロである理由: 引数の数が自由にできる。Rustの関数は引数の数が固定だが、マクロはコンパイル前にコード生成するので柔軟。

```rust
println!("hello");                    // 引数1つ
println!("{} + {} = {}", 1, 2, 3);    // 引数4つ
```

見分け方: 名前の末尾に `!` が付いていたらマクロ。

```rust
println!("...");     // マクロ
vec![1, 2, 3];       // マクロ（Vecを簡単に作る）
format!("...");      // マクロ（フォーマットしたStringを返す）
todo!();             // マクロ（未実装マーカー）
"hello".to_string(); // 関数（!がない）
```

マクロの種類:
- `macro_rules!` → 宣言的マクロ（パターンマッチでコード生成）
- `#[derive(Debug)]` → 手続き的マクロ（derive もマクロの一種）

Goとの比較: Goにはマクロがない。`go generate` やコード生成ツールで似たことをするが言語組み込みではない。

## Q6: マクロはコンパイル前に展開されて、最適なコードに変わってからコンパイルされる？

その通り。流れは: `ソースコード → [マクロ展開] → 展開後のコード → [コンパイル] → バイナリ`

ポイント:
- 型チェックはマクロ展開後に行われる（Display未実装ならコンパイルエラー）
- 実行時オーバーヘッドなし（展開後は普通のコードと同じ）
- Goの `go generate` と違い、ビルドプロセスに統合されているので手動実行不要

## Q7: `eprintln!` はログの出力レベルが違うということ？`cargo run` しても同じ表示に見える

ログレベルではなく、**出力先の違い**。

- `println!` → 標準出力（stdout）
- `eprintln!` → 標準エラー出力（stderr）

ターミナル上では同じに見えるが、リダイレクトで分離できる:
```bash
cargo run > output.txt  # stdout だけファイルへ。eprintln! はターミナルに残る
```

Goで言うと `fmt.Println`（stdout）と `fmt.Fprintln(os.Stderr, ...)`（stderr）の違い。正常な出力はstdout、エラーや診断情報はstderrに流すのがUnixの慣習。
