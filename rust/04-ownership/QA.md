# Q&A: 04-ownership

日付: 2026-03-06

## Q1: 関数に渡した後もsを使いたい場合、cloneするしかない？ヒープに同じものが2つ？returnしたらclone不要？

選択肢は3つ:

**1. clone** — ヒープに2つコピーが存在する。コストがかかる。
```rust
takes_ownership(s.clone());
println!("{s}");  // OK
```

**2. 関数から返す** — ヒープには1つだけ。所有者が移動するだけ。
```rust
fn takes_and_returns(s: String) -> String {
    println!("{s}");
    s
}
let s = takes_and_returns(s);
println!("{s}");  // OK
```

**3. 借用（05で学ぶ）** — 最もよく使われる方法。コピーもムーブもしない。
```rust
fn borrows(s: &String) {
    println!("{s}");
}
borrows(&s);
println!("{s}");  // OK
```

実際のRustコードでは3（借用）が最も一般的。所有権の不便さを解決するために借用が存在する。

補足: `println!` はマクロなので所有権を奪わない。所有権を奪うのは関数呼び出し。

## Q2: 構造体の中身がStringのとき、所有権はどう扱われる？

構造体が所有者になる。構造体がdropされると中のStringも一緒にdropされる。

```rust
struct User {
    name: String,
    age: u32,
}
let user = User { name: String::from("田中"), age: 30 };
let user2 = user;  // user 全体がムーブ（name が String なので Copy できない）
// user は使えなくなる
```

- フィールドを個別に取り出すとそのフィールドだけムーブされる
- `age`（i32、Copy）は取り出しても構造体に影響しないが、`name`（String）を取り出すと構造体全体が部分ムーブ状態になる
- Goでは構造体の値コピーで何も気にしなくてよいが、RustではStringを含む構造体はCopyできない

## Q3: `let user_name = user.name` したら user.name だけが所有権を失う？

その通り。**部分ムーブ（partial move）**と呼ばれる:
```rust
let user_name = user.name;      // name だけムーブ
println!("{}", user.age);       // OK（u32 は Copy）
// println!("{}", user.name);   // エラー！name はムーブ済み
// println!("{:?}", user);      // エラー！user 全体としては不完全
```

コンパイラがフィールド単位で所有権を追跡してくれる。Go にはない概念。

## Q4: 所有権の概念が言語レベルで導入された目的は？

大きく2つ:

**1. GCなしでメモリ安全を保証する**
- GC言語（Go等）→ 実行時コスト（停止時間、メモリ増加）がある
- C/C++ → GCなしで高速だが手動管理でバグの温床（二重解放、use-after-free）
- Rust → コンパイル時にメモリの寿命が全て決まる。実行時コストゼロで安全

**2. データ競合をコンパイル時に防止する**
- Goでは `go run -race` で検出できるが実行時チェック（実行されたパスしか検出不可）
- Rustでは不変参照は複数OK、可変参照は1つだけ、同時共存不可 → データ競合が原理的に起きない

**なぜ「言語レベル」か:**
解決方法はいくらでもあるが、人間はミスする。所有権は「守らないとコンパイルが通らない」レベルに引き上げた。人間の規律ではなくコンパイラが保証する仕組み。

ただしGoのGCは十分優秀なので、Webサービス等では所有権の学習コストに見合わないこともある。トレードオフ。

## Q5: GCがないのが理由？でも解放自体は自動でされるよね？

その通り、解放は自動。ただしGCとは仕組みが全く違う:

- Go（GC）→ 実行時に「もう使ってない？」を探索して解放。タイミング不定
- Rust（所有権）→ コンパイラが適切な位置に解放コードを挿入。スコープ終了で即解放。タイミング確定的
- C（手動）→ プログラマが `free()` を書く。忘れたらリーク

所有権は「いつ解放すべきか」をコンパイラが判断するためのルール。所有者が1つだけだからこそ、スコープを抜けたタイミングで安全に解放できる。実行時のGC探索が不要なのでオーバーヘッドがない。

## Q6: 演習1の解決として `String::from("hello")` を再度行っても良い？

良い。`clone` でも `String::from` を再度呼んでもどちらもヒープに "hello" が2つ存在するのでコスト的には同じ。ただし実際のコードでは同じ値を再度作るのは冗長なので `clone` の方が意図が明確。本当のベストは借用（05で学ぶ）。

## Q7: 演習2の回答確認

```rust
fn string_length(s: String) -> (String, usize) {
    let length = s.len();
    (s, length)
}
```
正解。所有権を返すためにタプルで返すパターン。ただしこの「所有権を返すためだけにタプルを使う面倒さ」が借用（05）で解消される。

## Q8: 演習3の回答確認

```rust
fn main() {
    let s = String::from("hello");
    let s = print_string(s);       // シャドウイングで再束縛
    let _ = print_string(s);       // 2回目
}

fn print_string(s: String) -> String {
    println!("{s}");
    s  // 所有権を返す
}
```
正解。ただし毎回所有権を返してもらうのは面倒。借用（05）で `&s` を渡せば何回でも呼べるようになる。
