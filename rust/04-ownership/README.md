# 04 - 所有権（Ownership）

## 概要

所有権はRust最大の特徴であり、**GC（ガベージコレクション）なしでメモリ安全を保証する仕組み**。Go経験者にとっては完全に新しい概念。

## 所有権の3つのルール

1. **各値には「所有者」となる変数が1つだけ存在する**
2. **所有者がスコープを抜けると、値は自動的に解放される**（drop）
3. **所有権は「ムーブ」によって別の変数に移る**

## Goとの比較

| 概念 | Go | Rust |
|---|---|---|
| メモリ管理 | GC（実行時に自動回収） | 所有権（コンパイル時に管理） |
| 変数の代入 | 常にコピー（参照型はポインタコピー） | ムーブまたはコピー（型による） |
| メモリ解放 | GCが判断 | スコープを抜けたら即座に解放 |
| 二重解放 | GCが防止 | 所有権ルールが防止 |
| 実行時コスト | GCの停止時間あり | なし |

### なぜGoにはなくてRustにはあるのか

```go
// Goのコード — 何も考えずにコピー・共有できる
s1 := "hello"
s2 := s1        // コピー（string は不変なので安全）
fmt.Println(s1) // OK

slice1 := []int{1, 2, 3}
slice2 := slice1          // 内部配列を共有（浅いコピー）
slice2[0] = 99
fmt.Println(slice1[0])    // 99（slice1も変わる！）
// → GCがあるのでメモリ管理の心配はない
// → ただしデータ競合のリスクはある
```

```rust
// Rustのコード — 所有権で安全性を保証
let s1 = String::from("hello");
let s2 = s1;        // ムーブ（s1 は使えなくなる）
// println!("{s1}"); // コンパイルエラー！
println!("{s2}");    // OK

// → 二重解放もデータ競合もコンパイル時に防止
```

## スタックとヒープ

所有権を理解するにはスタックとヒープの違いが重要。

| | スタック | ヒープ |
|---|---|---|
| サイズ | コンパイル時に確定 | 実行時に決まる |
| 速度 | 高速 | 低速 |
| 例 | `i32`, `bool`, `char`, `f64` | `String`, `Vec` |
| 代入時 | コピー（Copy トレイト） | ムーブ |

```
スタック上の i32:          ヒープ上の String:
┌─────────┐               ┌─────────┐     ┌───────────┐
│  x: 42  │               │ s1: ptr─┼────→│ "hello"   │
└─────────┘               │   len: 5│     └───────────┘
┌─────────┐               │   cap: 5│
│  y: 42  │ ← コピー      └─────────┘
└─────────┘
                           let s2 = s1; でムーブすると:
                           ┌─────────┐     ┌───────────┐
                           │ s2: ptr─┼────→│ "hello"   │
                           │   len: 5│     └───────────┘
                           │   cap: 5│
                           └─────────┘
                           s1 は無効化される（二重解放を防止）
```

Goではスライスの代入が浅いコピー（内部配列を共有）になるのと構造は似ているが、Rustはコンパイラが所有権を追跡して安全性を保証する。

## コード解説

### ムーブ

```rust
let s1 = String::from("hello");
let s2 = s1;  // 所有権が s2 にムーブ
// s1 はもう使えない
println!("{s2}"); // OK
```

### コピー（Copy トレイト）

```rust
let x = 42;
let y = x;  // コピー（i32 は Copy）
println!("{x} {y}"); // 両方使える
```

Copy される型:
- 整数型（`i32`, `u64` など）
- 浮動小数点型（`f32`, `f64`）
- `bool`
- `char`
- タプル（中身が全て Copy の場合）

### 関数と所有権

```rust
fn takes_ownership(s: String) {
    println!("{s}");
} // s はここで drop される

let s = String::from("hello");
takes_ownership(s);
// println!("{s}"); // コンパイルエラー！所有権は関数にムーブされた
```

```go
// Goでは何も気にせず渡せる
func takesString(s string) {
    fmt.Println(s)
}
s := "hello"
takesString(s)
fmt.Println(s) // OK（Goの string は不変で暗黙的にコピー）
```

### clone

```rust
let s1 = String::from("hello");
let s2 = s1.clone(); // ヒープデータの深いコピー
println!("{s1} {s2}"); // 両方使える
```

`clone()` は明示的にヒープデータを複製する。コストがかかるので意図的に書く必要がある。

## 演習

### 演習1: 基礎

以下のコードがコンパイルエラーになる理由を説明し、修正しよう。

```rust
fn main() {
    let s = String::from("hello");
    let t = s;
    println!("{s} {t}");
}
```

### 演習2: 応用

文字列を受け取り、長さと一緒に返す関数を作ろう。所有権を返すことで呼び出し元で引き続き使えるようにする。

```rust
fn string_length(s: String) -> (String, usize) {
    // ヒント: 長さを計算してから、s と長さをタプルで返す
}

fn main() {
    let s = String::from("hello, ownership!");
    let (s, len) = string_length(s);
    println!("{s} の長さは {len}");
}
```

### 演習3: チャレンジ

次のコードをコンパイルできるように修正しよう。ただし `clone()` は使わないこと。

```rust
fn main() {
    let s = String::from("hello");
    print_string(s);
    print_string(s); // 2回目の呼び出し
}

fn print_string(s: String) {
    println!("{s}");
}
```

ヒント: 関数のシグネチャを変えて所有権を返すようにする。次のトピック（05-borrowing）で、もっとスマートな解決方法を学ぶ。

## まとめ

- **所有権**はRust固有の概念。GCなしでメモリ安全を保証する
- 各値の所有者は**1つだけ**。スコープを抜けると自動的に **drop** される
- ヒープデータの代入は**ムーブ**（元の変数は使えなくなる）
- スタックデータは**コピー**（Copy トレイトを持つ型）
- 関数に値を渡すと所有権がムーブされる。返してほしければ戻り値で返す
- 明示的に複製したい場合は `.clone()` を使う
- 次のトピック（**05-borrowing**）で、所有権を移さずに値を使う方法を学ぶ
