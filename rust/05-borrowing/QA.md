# Q&A: 05-borrowing

日付: 2026-03-06

## Q1: `add_suffix(&mut s)` の例で不変借用はどこ？

不変借用はない。このコードは全て可変借用。不変借用（`&T`）と可変借用（`&mut T`）は「どちらかを選んで使う」もの。

- 読むだけ → `&T`（不変借用）
- 変更したい → `&mut T`（可変借用）

「同時に存在できない」ルールが効くのは、同じデータに対して両方を同時に持とうとしたとき:
```rust
let mut s = String::from("hello");
let r1 = &s;          // 不変借用
let r2 = &mut s;      // 可変借用 → コンパイルエラー（r1 がまだ生きている）
println!("{r1}");
```

## Q2: 「渡している間」っていつ？関数が終わったらアクセスできる？

その通り。関数呼び出しで `&mut` を渡した場合、呼び出しが返った瞬間に借用は終わる:
```rust
let mut s = String::from("hello");
add_suffix(&mut s);  // この呼び出しの間だけ借用
println!("{s}");      // 借用終了後なのでOK
```

変数に保持した場合は「最後に使われた場所」まで借用が続く（NLL）:
```rust
let mut s = String::from("hello");
let r = &mut s;       // 変数に保持 → 借用が続く
r.push_str("!");
println!("{r}");       // r の最後の使用 → ここで借用終了
println!("{s}");       // OK
```

## Q3: `let r = &v[0];` の不変借用はいつ解放される？

NLLのルールで `r` の最後の使用箇所で解放される:
```rust
let mut v = vec![1, 2, 3];
let r = &v[0];
println!("{r}");      // r の最後の使用 → ここで借用終了
v.push(99);           // OK（借用が終わった後）
```

順番を変えて `r` の使用前に `v.push` すると、借用がまだ生きているのでコンパイルエラーになる。

## Q4: `str` と `String` の違いは？

- `String` — ヒープ上の文字列。所有権あり。変更可能。可変長
- `&str` — 文字列への参照（スライス）。所有権なし。読み取り専用

Goで言うと `String` ≒ `strings.Builder`、`&str` ≒ `string`。

使い分け:
- データを持ちたい（構造体のフィールド等）→ `String`
- 読みたいだけ → `&str`
- 関数の引数 → `&str`（`String` も文字列リテラルも両方受け取れる）

## Q5: `&String` と `&str` の違いは？`&String` って何？

- `&String` — `String` 全体への参照
- `&str` — 文字列データの一部（または全体）へのスライス参照

`&String` は `&str` に自動変換されるので、`&String` を使う理由がほぼない。`&str` なら `String` も文字列リテラルも両方受けられるが、`&String` は文字列リテラルを受けられない。

```rust
fn f(s: &str) {}    // String も "literal" も受けられる
fn g(s: &String) {} // "literal" は受けられない
```

clippyも `&String` を使うと `&str` にしろと警告する。関数の引数は常に `&str` が慣習。

## Q6: `let s = "hello"` はできない？

できる。型は `&str`（文字列リテラル）になる:
```rust
let s = "hello";                // &str（バイナリに埋め込まれた静的データへの参照）
let s = String::from("hello");  // String（ヒープに確保）
```

- 変更しないなら `"hello"`（&str）
- 変更するなら `String::from("hello")`

Goは `string` が1種類だけだが、Rustは `&str` と `String` の2種類がある。

## Q7: `"hello"` と `String::from("hello")` の違い、ユースケースを詳しく

メモリの違い:
- `"hello"` → バイナリの静的領域に埋め込み。プログラム終了まで消えない。借用
- `String::from("hello")` → ヒープに確保。スコープを抜けると解放。所有

使い分け:
| 場面 | 型 | 理由 |
|------|------|------|
| 固定文字列・定数 | `&str` | 変更しない。ヒープ確保不要 |
| 関数の引数（読むだけ） | `&str` | 汎用的 |
| 実行時に組み立て・加工 | `String` | 変更が必要 |
| ユーザー入力 | `String` | サイズが事前に不明 |
| 構造体のフィールド | `String` | データを所有する必要がある |
| 関数の戻り値 | `String` | 所有権を渡す必要がある |

迷ったら「変更するか、所有する必要があるか」で判断。どちらもNoなら `&str`。

## Q8: `"hello"` はプログラム全体で使うstatic値？`&str` であって `str` ではない？

その通り。`"hello"` の正確な型は `&'static str`（プログラム全体で有効な静的データへの参照）。

`str` は直接持てない（サイズが不定）ので、常に `&str` として参照を通して使う:
```rust
// let s: str = "hello";   // エラー！サイズ不定
let s: &str = "hello";     // OK
```

Goの `string` も内部的にはポインタ+長さのペアで `&str` とほぼ同じ構造。Goはそれを1つの型に隠しているが、Rustは所有（`String`）と参照（`&str`）を明示的に区別している。

## Q9: 文字列リテラルがメモリ上に常にあってポインタだけ持つのは違和感がある。他の言語も同じ？

ほぼ全ての言語が同じ仕組み。文字列リテラルはバイナリのrodata（読み取り専用データ）セクションに埋め込まれ、変数はそこへのポインタを持つ。

- C → 隠さないが書き換えも止めない（危険）
- Go/Java/Python → 隠して安全にしている（不変 or GC管理）
- Rust → `&str`（借用）として型で明示。書き換えはコンパイル時に防止

違和感があるのは、他の言語が隠していたものをRustが型として見せているから。
