# 08 - トレイトとジェネリクス Q&A

日付: 2026-03-07

## Q1: トレイト実装の暗黙的（Go）と明示的（Rust）のメリット・デメリット

### 暗黙的（Go: ダックタイピング）
- **メリット**: 柔軟。他パッケージの型が後から定義したインターフェースを自動的に満たせる（`io.Reader` など）
- **デメリット**: 意図せず満たす事故がある。どのインターフェースを満たすかコードから読み取りにくい。リファクタリング時に気づきにくい

### 明示的（Rust: `impl Trait for Type`）
- **メリット**: 意図が明確。シグネチャ不一致は即コンパイルエラー。デフォルト実装を提供できる
- **デメリット**: 孤児ルール（Orphan Rule）による制約がある。ボイラープレートが増える

### 孤児ルール
「自分のクレートの型」か「自分のクレートのトレイト」のどちらかでないと `impl` できない。異なるクレートが同じ impl を定義して衝突するのを防ぐためのルール。

## Q2: 自分のトレイトを外部の型に実装するのはどういうとき？

既存の型に振る舞いを追加したいとき。**拡張トレイト（extension trait）** と呼ばれるパターン。

```rust
trait Stats {
    fn average(&self) -> f64;
}
impl Stats for Vec<i32> {
    fn average(&self) -> f64 { ... }
}
// vec![1, 2, 3].average() が使える
```

Goだと同じことをするにはラッパー型 `type IntSlice []int` を定義してキャストが必要。Rustでは `use` でトレイトをインポートするだけで既存の型にメソッドが追加される。`itertools` クレートが実例。

## Q3: `where T: Shape + ToJson + Debug` は3つ全てを満たす型だけ使える？

その通り。`+` で繋いだトレイトを全て実装している型でないとコンパイルエラー。Goだと埋め込みインターフェースを新たに定義する必要があるが、Rustは `+` で繋ぐだけ。

## Q4: トレイト境界をすり抜けるようなコードは書けるか？

書けない。Rustのトレイト境界はコンパイル時に厳格にチェックされる。同じメソッド名を持っていても `impl Trait for Type` を明示的に書かない限り、そのトレイトとして扱えない。Goだとたまたま同名メソッドを持つ型が意図せずインターフェースを満たしてしまうことがあるが、Rustではそれが起きない。

## Q5: HashMapのキーにcomparableじゃない型を使ったらどうなる？

コンパイルエラーになる（Go/Rust両方）。

- **Rust**: `HashMap` のキーには `Hash + Eq` トレイトが必要。derive していなければエラー。`f64` は `NaN != NaN` 問題のため `Hash` も `Eq` も実装しておらず、`f64` フィールドを持つ構造体は `Hash` を derive できない。
- **Go**: マップのキーには `comparable` が必要。スライスや map をフィールドに持つ構造体はキーにできない。

## Q6: PartialOrd と Ord の違いは？

`Ord` = ordering（順序）。"Partial" は数学の「半順序（partial order）」に由来する。

- **`PartialOrd`**: 比較できないペアがあってもよい。`partial_cmp()` は `Option<Ordering>` を返す（`None` = 比較不能）
- **`Ord`**: 全順序。任意の2値で必ず順序が決まる。`cmp()` は `Ordering` を返す（`None` なし）

典型例は `f64`。`NaN` はどの値とも順序が決まらないため `PartialOrd` のみ実装で `Ord` は未実装。そのため `Vec<f64>` に `.sort()` は使えない。整数型（`i32` 等）は `Ord` まで実装している。

同様に `PartialEq` vs `Eq` も同じ関係。`NaN != NaN` なので `f64` は `PartialEq` のみ。

Goではこの区別はなく、`float64` でも普通に `<` で比較できる（NaN の挙動は実質未定義）。Rustは数学的な正確さを型システムで表現している。

ジェネリクスでは必要最小限のトレイト境界を使うのが基本。比較するだけなら `PartialOrd`、ソートや `BTreeMap` のキーなど全順序が必要なら `Ord`。

## Q7: NaN を PartialOrd のジェネリック関数に入れるとどうなる？

コンパイルは通るが、結果がおかしくなる。NaN との比較は全て `false` になるため、`if a >= b` の分岐が引数の順序で結果が変わってしまう。

```rust
fn max_of<T: PartialOrd>(a: T, b: T) -> T {
    if a >= b { a } else { b }
}
max_of(f64::NAN, 1.0)  // → 1.0（たまたまOK）
max_of(1.0, f64::NAN)  // → NaN（おかしい）
```

`>=` は `bool` を返すだけで、比較不能を `false` にしてしまう。安全に扱うなら `partial_cmp()` を使い `Option<Ordering>` の `None`（比較不能）を明示的に処理する。あるいはトレイト境界を `Ord` にすれば `f64` はコンパイルエラーになり、全順序が必要な場面で浮動小数点を使わせない防御ができる。

## Q8: 静的ディスパッチのモノモーフィゼーションで実際に型ごとの関数が生成される？

はい。`fn print_area<T: Shape>(shape: &T)` を `Circle` と `Rectangle` で呼ぶと、バイナリには `print_area` が2つの別アドレスに生成される（`nm` コマンドで確認可能）。コンパイラが型ごとに専用コードをコピー生成し、各 `shape.area()` 呼び出しは具体型のメソッドへの直接呼び出しになる。インライン化もされやすい。代償としてバイナリサイズが増える。

Goのインターフェースは常にvtable経由の動的ディスパッチで関数は1つだけ。Rustは静的（ジェネリクス）と動的（`dyn Trait`）を選べるのがポイント。

## Q9: モノモーフィゼーションでバイナリが膨張する？

する。型パラメータ2つ × 5型ずつで25個の関数が生成される。`dyn Trait`（動的ディスパッチ）版と比較するとtextセクションが数十KB増える。型パラメータの掛け算やジェネリック関数の連鎖でコピーが増殖し、`serde` や `futures` のようなクレートで実際にコンパイル時間・バイナリサイズが問題になることがある。対策は型に依存しないロジックを非ジェネリック関数に切り出し、ジェネリック部分を最小限にすること。Goのインターフェースは常に動的ディスパッチなのでこの問題は起きない。

## Q10: コンパイル時に型が確定しない場合（ユーザー入力など）はどうする？

`dyn Trait`（動的ディスパッチ）を使う。`Box<dyn Shape>` のように返せば、実行時の入力に応じて異なる具体型を返せる。呼び出し側は具体型を知らずにトレイトのメソッドだけで操作する。Goのインターフェースと同じ発想。

`match` の網羅性はコンパイラが強制するが（`&str` には `_` が必要）、対応する型の網羅は設計次第でデフォルト値やエラーを返せばよい。Rustは静的（ジェネリクス）と動的（`dyn Trait`）を場面に応じて使い分けられるのがGoとの違い。

## Q11: derive が使えない（失敗する）ケースはある？エラーはいつ出る？

ある。`derive` はマクロで機械的にコードを展開するだけなので、**全フィールドがそのトレイトを実装していないとコンパイルエラー**になる。

### エラーが出るタイミング

**コンパイル時**。`derive` はプロシージャルマクロで、コンパイルの初期段階でコードに展開される。例えば `#[derive(Clone)]` は概ねこう展開される:

```rust
impl Clone for MyStruct {
    fn clone(&self) -> Self {
        MyStruct {
            field1: self.field1.clone(),
            field2: self.field2.clone(),
        }
    }
}
```

この展開結果に対して通常の型チェックが走り、フィールドの型が `Clone` を実装していなければその時点でエラーになる。つまり `derive` 自体が特別なチェックをしているわけではなく、展開後のコードが型チェックに通らないだけ。

### 典型的な失敗パターン

| derive したいトレイト | 失敗する原因の例 |
|---|---|
| `Clone` | フィールドに `Clone` 未実装の型がある（例: `&mut T`） |
| `Copy` | フィールドに `Copy` 未実装の型がある（`String`, `Vec<T>` などヒープ確保する型） |
| `Eq` / `Ord` | `f32`, `f64` は `NaN` があるため `Eq`/`Ord` 未実装 |
| `Hash` | `f32`, `f64` は `Hash` 未実装 |
| `Default` | フィールドに `Default` 未実装の型がある |

### 対処法

derive できない場合は手動で `impl` を書く:

```rust
struct MyStruct {
    value: f64,
}

// f64 は Eq を derive できないが、NaN が来ないと保証できるなら手動実装できる
impl Eq for MyStruct {}
```

### Goとの比較

Goのインターフェースは「メソッドがあれば自動的に満たす」ので、derive のような「一括導出が失敗する」という概念自体がない。Rustはコンパイル時に厳密にチェックするからこそ、こうした制約が表面化する。

## Q12: ライフタイムパラメータ `'a` とは？

関数シグネチャに現れる `'a` は**ライフタイムパラメータ**。参照の有効期間に名前を付けたもの。

```rust
fn longest<'a, T>(a: &'a [T], b: &'a [T]) -> &'a [T]
```

- `&'a [T]` = ライフタイム `'a` の間だけ有効なスライスへの参照
- 戻り値も `&'a [T]` なので、「返す参照は引数 a と b のどちらも生きている間だけ有効」とコンパイラに伝えている
- 参照を返す関数ではコンパイラが「返された参照がいつまで有効か」を知る必要があるため、ライフタイム注釈が必要

`Formatter<'_>` の `'_` は「ライフタイムを推論に任せる」という省略記法。

**Goとの比較**: GoにはGCがあるため参照の有効期間を気にする必要がなく、ライフタイムの概念がない。Rustはランタイムコストなしで安全性を保証するために、コンパイル時にライフタイムを検証する。

※ ライフタイムの詳細はカリキュラムのトピック11で扱う。演習2のシグネチャでは先取りして登場している。

## Q13: 動的ディスパッチ、Vec、vtable とは？

### Vec

`Vec` は可変長の配列。Goのスライスに相当する。

```rust
let v: Vec<i32> = vec![1, 2, 3];  // Rust
// v := []int{1, 2, 3}             // Go
```

### vtable（仮想関数テーブル）

動的ディスパッチの仕組みの核心。`&dyn Shape` は内部的に2つのポインタを持つ:

1. **データへのポインタ**（実際の Circle や Rectangle）
2. **vtable へのポインタ**（どのメソッドを呼ぶかの表）

vtable は型ごとに1つ生成される:

```
vtable (Circle用)
┌──────────────┬────────────────────┐
│ area         │ → Circle::area     │
│ perimeter    │ → Circle::perimeter│
│ describe     │ → Shape::describe  │
└──────────────┴────────────────────┘
```

メソッド呼び出し時に vtable を参照して正しい関数を見つけて呼ぶ。

### 静的 vs 動的ディスパッチ

- **静的**（`impl Shape`）: コンパイル時に型が確定。直接呼び出しで高速
- **動的**（`dyn Shape`）: 実行時に vtable 経由で解決。異なる型を1つのコレクションに混在できる

```rust
// Circle と Rectangle を1つの Vec に入れられる
let shapes: Vec<&dyn Shape> = vec![&circle, &rect];
```

### Goとの比較

Goのインターフェース変数も内部的に `(データポインタ, itable)` の2ワード構成で、仕組みはまったく同じ。ただしGoは**常に動的ディスパッチ**で選択の余地がない。Rustは静的と動的を場面に応じて選べる。
