# 11 - ライフタイム

## 概要

ライフタイム (lifetime) は **「ある参照が有効である範囲」を型システムに乗せる仕組み**。借用チェッカーがダングリング参照（解放済みデータを指す参照）をコンパイル時に検出するために使う。

GC を持つ Go では一切意識する必要がない概念だが、Rust では「参照を返す関数」「参照を持つ構造体」を書く瞬間に必ず登場する。Go 経験者にとって最も馴染みのない領域。

## Goとの比較

| 概念 | Go | Rust |
|---|---|---|
| メモリ解放 | GCが実行時に判定 | スコープ終了で即解放 |
| ダングリングポインタ | 起きない (GCが防ぐ) | 起きる可能性 → 借用チェッカーで防ぐ |
| 参照の生存期間 | 暗黙 (GCに任せる) | 明示的に型で表現 |
| 関数の戻り値が参照 | `*T` を返してOK (GCが追跡) | ライフタイム注釈が必要なことがある |
| 構造体に参照フィールド | `*T` でOK | ライフタイム注釈が必須 |

### Go で書いた場合

```go
// Go — GC があるのでダングリング参照は発生しない
func longest(x, y string) string {
    if len(x) > len(y) {
        return x
    }
    return y
}

type Excerpt struct {
    part string  // 値型として保持 (またはポインタ)
}
```

### Rust で書く場合

```rust
// Rust — ライフタイム注釈で「戻り値の寿命は入力に紐づく」と明示
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}

struct Excerpt<'a> {
    part: &'a str,  // 参照を持つ場合は注釈必須
}
```

主な違い:
- **Goは「いつ解放されるか」を考えなくて良い**。コンパイル時にもランタイムにも自動で管理される
- **Rustは参照の寿命を型として書く**。書かない場合は省略規則 (後述) が適用されるか、コンパイルエラー
- ライフタイムは**実行時の挙動を変えない**。型検査のためだけの情報 (ゼロコスト)

## コード解説

### なぜ必要か

```rust
let r;
{
    let x = 5;
    r = &x;       // x は内側のブロックで解放される
}
println!("{r}");  // 解放済みのデータを参照 → コンパイルエラー
```

借用チェッカーは「`r` のライフタイムは少なくとも `println!` まで必要」「`x` のライフタイムはブロック内まで」と推論し、矛盾を検出する。

### ライフタイム注釈の構文

```rust
&i32        // 参照
&'a i32     // 明示的なライフタイム 'a を持つ参照
&'a mut i32 // 可変参照 + ライフタイム
```

`'a` (ティック a) はジェネリックなライフタイムパラメータで、ジェネリック型 `<T>` と同じく `<'a>` で導入する。

```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

意味:
- `<'a>` でライフタイムパラメータを宣言
- `x: &'a str`, `y: &'a str` — どちらも `'a` 以上生きる参照
- 戻り値 `&'a str` も `'a` の範囲で有効

**実際の `'a` は呼び出し側で決まる**: `x` と `y` の生存期間の**共通区間**（両方が生きている範囲）が `'a` になる。だから戻り値は「短い方の寿命」に律速される。

```rust
let s1 = String::from("long");
{
    let s2 = String::from("short");
    let r = longest(&s1, &s2);
    // r は s1, s2 のうち短い方 = s2 の寿命まで有効
    println!("{r}");
}
// ここでは s2 が解放されているので r は使えない
```

### ライフタイム省略規則 (Lifetime Elision)

頻出パターンでは注釈を書かなくて良い。コンパイラが以下の3規則で補う:

| 規則 | 内容 |
|---|---|
| 1 | 入力の各参照に **異なる** ライフタイムを自動付与 |
| 2 | 入力ライフタイムが**1つだけ**なら、それを全ての出力に割り当てる |
| 3 | メソッド (`&self` を持つ) なら、**`self` のライフタイム**が出力に割り当てられる |

### 例: `first_word`

```rust
fn first_word(s: &str) -> &str { ... }
// ↑ 規則2 が適用され、実体は↓
fn first_word<'a>(s: &'a str) -> &'a str { ... }
```

### 例: `longest`

```rust
fn longest(x: &str, y: &str) -> &str { ... }
// 規則1 で 'a, 'b が付くが、戻り値にどちらを割り当てるか不明
// → コンパイルエラー。明示が必要
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str { ... }
```

### 例: メソッド

```rust
impl<'a> Excerpt<'a> {
    fn announce(&self, announcement: &str) -> &str {
        // 規則3 で &self のライフタイムが戻り値に
        self.part
    }
}
```

### 構造体に参照を持たせる

```rust
struct Excerpt<'a> {
    part: &'a str,
}
```

意味: **「`Excerpt` のインスタンスは `part` が指す元データより長く生きられない」** という制約。

```rust
let excerpt;
{
    let s = String::from("hello world");
    excerpt = Excerpt { part: &s };
}
// println!("{}", excerpt.part);  // コンパイルエラー: s は解放済み
```

Go では `string` は値型 (`[]byte` の不変版) なので所有権の問題は起きない。Rust で `&str` を構造体に入れる場合は必ずライフタイムを書く。

### `'static` ライフタイム

プログラム全体で生きる参照を表す特別なライフタイム。

```rust
let s: &'static str = "hello";  // 文字列リテラルはバイナリに埋め込まれる
```

**注意**: 「とりあえず `'static` を書けば通る」はアンチパターン。本当に必要な場面 (定数、グローバル) でのみ使う。多くの場合、設計を見直して借用を別の形にすべき。

### ライフタイムはランタイムに影響しない

```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str { ... }
```

このコードは最適化後、ライフタイム注釈を全て取り除いたコードと**完全に同じ機械語**になる。あくまでコンパイル時の検査のための情報。Go の GC のような実行時オーバーヘッドはない。

## 演習

### 演習1: 基礎

2つの `&str` のうち、**短い方**を返す関数を作ろう。

```rust
fn shortest<'a>(x: &'a str, y: &'a str) -> &'a str {
    todo!()
}
// shortest("hello", "hi") => "hi"
```

ヒント: `longest` をひっくり返すだけ。`'a` の意味を改めて考えてみよう。

### 演習2: 応用

文字列を受け取り、**最初の単語**と**残り**をタプルで返す関数を作ろう。

```rust
fn split_first_word(s: &str) -> (&str, &str) {
    todo!()
}
// split_first_word("hello world foo") => ("hello", "world foo")
// split_first_word("hello")           => ("hello", "")
```

ヒント:
- スペースの位置をループで探し、`&s[..i]` と `&s[i+1..]` を返す
- 省略規則1 と 2 を考えると、なぜこのシグネチャでコンパイルが通るのか？
  → 入力が1つなのでその寿命が両方の戻り値に伝搬する

### 演習3: チャレンジ

テキストとその中の**最も長い単語への参照**を保持する構造体を作ろう。

```rust
struct Highlight<'a> {
    text: &'a str,
    part: &'a str,
}

impl<'a> Highlight<'a> {
    fn new(text: &'a str) -> Self {
        todo!()
    }
}
```

ヒント:
- `text.split_whitespace()` で単語のイテレータが得られる (10章)
- `max_by_key(|w| w.len())` で最長を取得
- 戻り値 `Self` は `Highlight<'a>` の省略形

考察ポイント:
- なぜ `text` と `part` が同じ `'a` で良いのか?
  → `part` は `text` 内のサブスライスなので、`text` が生きている間は `part` も生きている。型としては「両方とも 'a 以上」で十分

## まとめ

- **ライフタイムは参照の生存期間をコンパイル時に検査する仕組み**。実行時コストはゼロ
- **省略規則** で多くの場合は注釈を書かなくて済む。複数の入力参照を持つ関数で初めて明示が必要
- **構造体に参照を持たせる場合は必ず注釈** が要る (`struct Foo<'a> { x: &'a T }`)
- **`'a` は呼び出し側で決まる**。複数の参照に同じ `'a` を付けると「共通区間」になる
- **`'static`** は全期間有効。文字列リテラルなど限定的に使う
- Go は GC のためライフタイムを意識しないが、Rust では「参照を返すならライフタイム」と心得る
