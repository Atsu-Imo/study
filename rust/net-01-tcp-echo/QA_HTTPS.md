# HTTPS Q&A

日付: 2026-05-21

ネットワーク学習の延長として、HTTPS にまつわる質問とその回答を記録する。
TLS、証明書、PKI、ハンドシェイク、暗号スイート、HSTS、HTTP/2/3 など。

## 登場人物

メイン:
- **クライアント** — ブラウザ、curl、モバイルアプリ、Rust の reqwest など
- **サーバー** — nginx, Apache, axum, hyper など

サブ:
- **CA (認証局)** — サーバー証明書を発行する第三者
- **OS 信頼ストア** — クライアント側でルート CA を保管する場所 (`/etc/ssl/certs/` 等)
- **DNS サーバー** — TLS 開始前の名前解決で登場
- **中間プロキシ / ロードバランサー** — TLS Termination 構成で登場

クライアント / サーバー内部の構造:
```
┌─ クライアント or サーバー ─────────┐
│  アプリ (curl, ブラウザ, nginx等)  │
│    └─ TLS ライブラリ              │
│  ── syscall ───────────────────   │
│  カーネル (TCP/IP)                │
│  NIC                              │
└────────────────────────────────────┘
```

## Q1: HTTPS の仕組みについてざっくり教えて

HTTPS = **HTTP を TLS で暗号化したもの**。TLS が下回りで頑張ってくれて、上では普通の HTTP が走る。

### レイヤー構造

```
[ HTTP (普通のリクエスト/レスポンス) ]   ← アプリ層
            ↓
[ TLS (暗号化・認証レイヤー) ]            ← ★ HTTPS の正体
            ↓
[ TCP (信頼性のある転送) ]                ← net-01 でやった層
            ↓
[ IP / NIC / カーネル / ... ]
```

TCP コネクションを張った後、その上で**まず TLS ハンドシェイクをやって暗号化トンネルを作る**。それから HTTP を送る。HTTP の中身は何も変わらない。

### TLS が提供する3つの保証

| 保証 | 何を防ぐか | 仕組み |
|---|---|---|
| 機密性 (Confidentiality) | 盗聴 | 通信内容を暗号化 |
| 完全性 (Integrity) | 改ざん | MAC (メッセージ認証コード) で検証 |
| 認証 (Authenticity) | なりすまし | サーバー証明書で本物確認 |

HTTP の弱点 (平文・なりすまし可能) を全部解決する。

### ハンドシェイクの流れ

```
クライアント                           サーバー
    │                                    │
    │── (1) TCP 3ウェイハンドシェイク ──→│  ← まず普通の TCP 接続
    │                                    │
    │── (2) ClientHello ────────────────→│  「対応する暗号スイート、ランダム値」
    │←── (3) ServerHello + 証明書 ──────│  「使う暗号、サーバーの証明書」
    │                                    │
    │       [証明書を CA で検証]         │
    │                                    │
    │── (4) 鍵交換 ─────────────────────→│  共通の暗号鍵を作る
    │←── (5) Finished ───────────────────│
    │                                    │
    │═══ (6) ここから暗号化通信 ════════│
    │── 暗号化された HTTP リクエスト ───→│
    │←── 暗号化された HTTP レスポンス ───│
```

ステップ 2-5 が「ハンドシェイク」。終わると暗号化トンネルが完成。

### 鍵の仕組み — 対称暗号と非対称暗号の使い分け

| 種類 | 例 | 特徴 | 使う場面 |
|---|---|---|---|
| 対称暗号 | AES, ChaCha20 | 高速。ただし鍵共有が必要 | 本番のデータ転送 |
| 非対称暗号 (公開鍵暗号) | RSA, ECDSA | 低速。公開鍵で暗号化→秘密鍵で復号 | 鍵交換、署名 |

現代の TLS は「**非対称で対称の鍵を安全に共有**」してから、本データは対称で高速に暗号化。具体的には ECDHE (楕円曲線ディフィー・ヘルマン鍵交換) で双方が同じセッション鍵を計算し、以降は AES などで通信する。

### 証明書と PKI

サーバー証明書の中身:
- サーバーのドメイン名 (例: `amazon.com`)
- サーバーの公開鍵
- 発行者 (認証局, CA) の署名

クライアントの検証:
1. 証明書の署名を CA の公開鍵で検証 (CA の公開鍵はブラウザ/OS に組み込み済み)
2. ドメインが一致するか
3. 有効期限内か
4. 失効していないか (OCSP/CRL)

この信頼の連鎖 = **PKI (Public Key Infrastructure)**。

```
[ ルート CA ] ← ブラウザ/OS に最初から信頼済み
     ├─ 署名
     ↓
[ 中間 CA ] (例: Let's Encrypt R3)
     ├─ 署名
     ↓
[ サーバー証明書 ] (example.com 用)
```

Let's Encrypt は中間 CA を運営して誰でも無料で証明書を発行できるようにしている。

### TLS 1.2 と 1.3 の違い

現代は **TLS 1.3** が主流 (2018 標準化)。

| 項目 | TLS 1.2 | TLS 1.3 |
|---|---|---|
| ハンドシェイク往復 | 2 RTT | 1 RTT (再接続なら 0 RTT も可能) |
| 暗号スイート | 多種多様、危ないものも残存 | 安全なものに絞り込み |
| 鍵交換 | RSA も使える (前方秘匿性なし) | 必ず DHE/ECDHE (前方秘匿性あり) |
| ハンドシェイク内容 | 一部平文 | ほぼ全部暗号化 |

**前方秘匿性 (Forward Secrecy)** — 将来サーバーの秘密鍵が漏れても過去の通信は解読できない性質。TLS 1.3 では必須。

### HTTPS アクセスの全体タイムライン

```
1. DNS で example.com の IP を引く (まだ HTTP/HTTPS じゃない)
2. TCP 接続を張る (3ウェイハンドシェイク)
3. TLS ハンドシェイク (証明書検証 + 鍵交換)
4. HTTP リクエスト送信
5. HTTP レスポンス受信
6. TLS で復号して中身を見る
7. TCP 切断
```

### HTTP/2, HTTP/3 との関係

- HTTP/2 は TLS の上で動く HTTP の新バージョン。ヘッダ圧縮、多重化
- HTTP/3 は TCP ですらなく **QUIC** (UDP ベース) の上で動く。TLS 1.3 を QUIC に内蔵
- どちらも実質 HTTPS 必須

### 要点

- HTTPS = HTTP + TLS。TCP の上に TLS の層を挟む
- TLS は機密性・完全性・認証の3つを保証
- 流れ: TCP 接続 → TLS ハンドシェイク (証明書検証 + 鍵交換) → 暗号化された HTTP 通信
- 非対称暗号で鍵共有、対称暗号で本通信、というハイブリッド
- 証明書は CA の署名チェーンで検証 (PKI)
- TLS 1.3 が現代の標準。1 RTT、前方秘匿性必須、安全なスイートのみ

## Q2: ハンドシェイクってどういう動作? HTTPS のリクエストは見た目1往復に見えるが、内部では違う?

**見た目は1往復、実際は数往復**。HTTP がステートレスなのはアプリ層の話で、下層 (TCP/TLS) は完全にステートフル。

### ハンドシェイクとは

本通信の前にお互いの状態をすり合わせる準備フェーズ。

- 「あなたはどのバージョン使えますか?」
- 「どの暗号方式に対応してますか?」
- 「鍵を交換しましょう」
- 「本物だと証明できますか?」
- 「OK、準備完了」

TCP の3ウェイハンドシェイクも同じ概念。TLS は TCP の上でさらに暗号化のすり合わせをする。

### TLS 1.3 ハンドシェイクの中身 (1 RTT)

#### ステップ1: ClientHello (クライアント → サーバー)

```
ClientHello {
    対応する TLS バージョン: [TLS 1.3, TLS 1.2]
    対応する暗号スイート: [TLS_AES_128_GCM_SHA256, ...]
    クライアントランダム値: 32バイトの乱数
    鍵交換用の公開鍵 (Key Share): 楕円曲線上の点
    アクセスしたいドメイン (SNI): "example.com"
}
```

「私はこれだけのカードを持っています、どれで行きましょう?」という申告。

#### ステップ2: ServerHello + 証明書 + Finished (サーバー → クライアント)

```
ServerHello {
    選択した暗号スイート: TLS_AES_128_GCM_SHA256
    サーバーランダム値: 32バイトの乱数
    鍵交換用の公開鍵 (Key Share)
}
[ここから暗号化]
Certificate { サーバー証明書チェーン }
CertificateVerify { 「この秘密鍵は私のもの」を証明する署名 }
Finished { ハンドシェイクの HMAC }
```

サーバーは: 暗号スイートを選び、Key Share を返し (これで両者が同じセッション鍵を計算可能)、証明書を送り、秘密鍵所有を署名で証明、改ざんなしを HMAC で証明。

#### ステップ3: クライアント検証 + Finished

クライアントは:
1. 証明書チェーンを CA で検証
2. ドメイン名と SAN が一致するか確認
3. 有効期限・失効を確認
4. CertificateVerify の署名を検証
5. 自分でもセッション鍵を計算
6. `Finished` を暗号化して返す

これで暗号化トンネル完成。

### 鍵交換の数学的マジック (DH)

「公開情報の交換だけ」でお互いが同じ秘密鍵を計算できる:

```
クライアントの秘密値 a, 公開値 g^a
サーバーの秘密値 b, 公開値 g^b

→ クライアント: g^b に a を掛ける → g^(ab)
→ サーバー: g^a に b を掛ける → g^(ab)

→ 両者とも g^(ab) を持つ! 傍受者には a も b も分からない
```

(実際は楕円曲線上の演算)

### HTTPS リクエストの実際の往復数

`curl https://example.com/` を新規接続で実行:

| フェーズ | 往復数 |
|---|---|
| DNS 解決 | 1 RTT |
| TCP 3ウェイハンドシェイク | 1 RTT |
| TLS 1.3 ハンドシェイク | 1 RTT |
| HTTP リクエスト/レスポンス | 1 RTT |
| 合計 | **約 4 RTT** |

東京-US で RTT 150ms なら合計 600ms。1回のリクエストでも内部では4往復。

### Wireshark で見るとどう見えるか

```
1. DNS query           → DNS サーバー
2. DNS response        ← DNS サーバー
3. TCP SYN             → サーバー
4. TCP SYN+ACK         ← サーバー
5. TCP ACK             → サーバー
6. ClientHello         → サーバー
7. ServerHello + Cert  ← サーバー
8. Finished            → サーバー
9. GET /index.html     → サーバー (暗号化)
10. 200 OK             ← サーバー (暗号化)
11. FIN                → サーバー
12. FIN+ACK            ← サーバー
```

表面では1往復に見えるが、実際は10往復以上のパケット。

### 「ステートレス」の正体 — レイヤーで分ける

| レイヤー | ステート | 何を保持しているか |
|---|---|---|
| HTTP (アプリ層) | **ステートレス** | 各リクエストは独立。前を覚えていない |
| TLS | **完全にステートフル** | セッション鍵、暗号スイート、シーケンス番号 |
| TCP | **完全にステートフル** | 接続状態、シーケンス番号、ウィンドウサイズ |
| IP | ステートレス | 各パケットは独立 |

「HTTP がステートレス」は**アプリレベル**の話。Cookie やセッション ID がない限りサーバーは前のリクエストを覚えていない。**通信路 (TCP+TLS) はステートフル**。

### 接続を使い回す仕組み

毎リクエストでハンドシェイクは非効率なので:

#### HTTP/1.1 Keep-Alive
同じ TCP+TLS 接続で複数の HTTP リクエスト/レスポンスを順次流す。

```
ハンドシェイク → GET /a → 200 → GET /b → 200 → ... → 切断
```

クライアントが1ページ分のリクエストを送ると数十リクエスト飛ぶが、ハンドシェイクは最初の1回だけ。

#### HTTP/2 多重化 (Multiplexing)
1接続上で複数リクエストを並列に流す。

```
ハンドシェイク → GET /a, GET /b, GET /c 並列 → 200 が並列に返る
```

#### TLS セッション再開 (Session Resumption)
- Session ID: サーバーが過去セッションを記憶
- Session Ticket: クライアントが暗号化チケットを保管
- TLS 1.3 の **0-RTT**: 再接続時にハンドシェイクと同時に最初の HTTP を送れる

### Cookie/セッション ID — ステートレス HTTP の上でステートを作る

HTTP 自身はステートレスなので、ログイン状態の保持にはアプリレベルの工夫:

- Cookie: クライアントがサーバーから受け取った値を保管、次回送り返す
- セッション ID: Cookie 内にランダム ID、サーバーが ID→ユーザー を紐づけ
- JWT: トークン自体にユーザー情報を埋め込む

「ステートレスな HTTP の上で、アプリレベルで状態を作る」工夫。

### curl で観察

```bash

* TLSv1.3 (OUT), TLS handshake, Client hello (1):

```

### 要点

- ハンドシェイク = 本通信前にお互いの状態をすり合わせる準備フェーズ
- TLS 1.3 のハンドシェイクは ClientHello → ServerHello+Cert+Finished → Finished の1往復
- 内部では鍵交換 (DH)・証明書検証・HMAC 検証が行われる
- 1回のリクエストの裏に DNS+TCP+TLS+HTTP で約4往復
- HTTP がステートレスなのはアプリ層の話。TCP/TLS は完全にステートフル
- 効率化のため Keep-Alive、HTTP/2 多重化、TLS セッション再開でハンドシェイク削減
- Cookie/セッション ID は「ステートレス HTTP の上でアプリがステートを作る」工夫

## Q3: 非対称暗号で「クライアントが公開鍵で暗号化、サーバーが秘密鍵で復号」という認識で合っている? 公開鍵はどこにあるの?

**TLS 1.2 までは合っていたが、TLS 1.3 では違う**。重要な分かれ目。

### TLS 1.2 までの RSA 鍵交換 (古い方式)

```
1. クライアントが乱数 (premaster secret) を生成
2. サーバーの公開鍵で暗号化して送る
3. サーバーは秘密鍵で復号
4. 両者がこの premaster secret からセッション鍵を導出
5. 以降は対称暗号で通信
```

この方式の致命的な問題: **前方秘匿性がない**。サーバーの秘密鍵が将来漏れたら、過去にキャプチャされた全通信が解読される。Snowden 事件以降この問題が重視され、TLS 1.3 では禁止。

### TLS 1.3 の現代的なやり方 — ECDHE 鍵交換

公開鍵で暗号化は使わず、ECDHE (楕円曲線ディフィー・ヘルマン) で鍵を共有:

```
1. クライアントが使い捨て鍵ペア (a_priv, a_pub) を生成
2. サーバーも使い捨て鍵ペア (b_priv, b_pub) を生成
3. 公開部分だけ交換
4. クライアント: a_priv と b_pub から共有秘密 S を計算
5. サーバー: b_priv と a_pub から同じ S を計算
6. S からセッション鍵を導出
7. 以降は対称暗号で通信
```

- a_priv, b_priv は**1セッション限りで捨てる** (ephemeral)
- 証明書の鍵とは完全に別物
- 証明書の秘密鍵が将来漏れても過去の S は再現不可能 → **前方秘匿性**

### 証明書の公開鍵は何のため?

**署名 (Signature)** のため。

```
1. サーバーが ECDHE 用の使い捨て公開鍵 b_pub を送る
2. ★ サーバーが「この b_pub は本当に私が作った」を
    証明書の秘密鍵で署名する (CertificateVerify)
3. クライアントは証明書の公開鍵で署名を検証
4. 「証明書の本物の所有者がこの b_pub を送ってきた」が確認できる
```

公開鍵暗号アルゴリズムは2つの用途があり、TLS 1.3 では署名側しか使わない:

| 用途 | 流れ | 何のため |
|---|---|---|
| 暗号化 | 公開鍵で暗号化 → 秘密鍵で復号 | 機密性 |
| 署名 | **秘密鍵で署名 → 公開鍵で検証** | 認証 |

### 公開鍵はどこにあるか

**サーバーが送ってくる証明書の中**。事前配布ではなく、接続のたびに送られる。

```
ハンドシェイク (TLS 1.3):

1. ClientHello → サーバー
2. ServerHello + Certificate ← サーバー
   - ★ ここで証明書 (公開鍵入り) が届く
3. クライアントが証明書の公開鍵で署名を検証
4. CA の署名で証明書自体を検証 (チェーン検証)
```

### 公開鍵は公開して安全か

そう。公開鍵で**暗号化はできるが復号はできない**、**署名検証はできるが署名生成はできない**。「秘密鍵だけが秘密」が原則。

```bash
# 任意サーバーの証明書 (公開鍵入り) を取得して中身を見る
$ openssl s_client -connect example.com:443 -showcerts
$ echo | openssl s_client -connect example.com:443 2>/dev/null | openssl x509 -text
```

### 信頼チェーン

```
[ ルート CA ]    ← ブラウザ/OS に最初から組み込み済み
    │ 署名
    ↓
[ 中間 CA ]      ← サーバーが送ってくる
    │ 署名
    ↓
[ サーバー証明書 ]  ← サーバーが送ってくる、公開鍵入り
```

ルート CA の公開鍵だけが事前配布。残りは動的に届く。

```bash
# Linux のルート CA 一覧
$ ls /etc/ssl/certs/
```

### TLS 1.3 における鍵の役割整理

```
証明書の鍵ペア (RSA/ECDSA)
  ├ 用途: 本人確認の署名
  └ 寿命: 数ヶ月〜数年

ECDHE の鍵ペア (使い捨て)
  ├ 用途: 共有秘密 S の素材
  └ 寿命: 1セッションだけ

セッション鍵 (対称、AES など)
  ├ 用途: HTTP 本文の暗号化
  ├ 由来: S から導出
  └ 寿命: 1セッションだけ
```

### 要点

- TLS 1.2 までは「公開鍵で premaster secret を暗号化」が正しい (前方秘匿性なし、現代は禁止)
- TLS 1.3 では公開鍵は**署名検証のためだけ**に使う
- セッション鍵は ECDHE で両者が計算する (鍵を送り合わない)
- 公開鍵はサーバーが証明書として接続時に送ってくる (事前配布ではない)
- ルート CA の公開鍵だけはブラウザ/OS に組み込み、信頼の起点
- 公開鍵は誰でも見て OK。秘密鍵だけが秘密

## Q4: 対称暗号と非対称暗号はどう使い分けてる? 理由も知りたい

**両方とも単独では実用にならず、組み合わせる必要がある**。それぞれに強みと弱みがあるので。

### それぞれの強みと弱み

| 観点 | 対称暗号 (AES, ChaCha20) | 非対称暗号 (RSA, ECDSA, ECDHE) |
|---|---|---|
| 速度 | 超速い | 超遅い |
| 鍵の数 | 1つ (両者で共有) | 2つ (公開鍵+秘密鍵のペア) |
| 鍵の配布 | 「どう共有するか」が大問題 | 秘密鍵を渡す必要なし |
| 適する用途 | 大量データの暗号化 | 鍵交換、署名、認証 |
| 暗号化できるサイズ | 任意 | 鍵長に依存 (RSA-2048 で 245 バイト程度) |

### 速度差の具体的な数字

| アルゴリズム | スループット |
|---|---|
| AES-256-GCM (対称) | **5〜10 GB/秒** (AES-NI 使用) |
| ChaCha20-Poly1305 (対称) | 1〜3 GB/秒 |
| RSA-2048 暗号化 | 〜100 MB/秒 |
| RSA-2048 復号 (非対称) | **〜2 MB/秒** ← 対称の数千倍遅い |
| ECDSA-P256 署名生成 | 〜1 MB/秒 |
| ECDHE-P256 鍵交換 | 〜数十μ秒/回 |

全データを非対称で暗号化したら、Web レスポンスが数千倍遅くなる。

### それぞれが解決する問題

#### 対称暗号の問題: 鍵配布問題

「同じ鍵を両者が持つ」のは速度の代償だが、その鍵をどう渡すか?
- ネットワーク経由で送る → 盗聴される
- 事前に物理で渡す → スケールしない

第二次世界大戦のエニグマ機もこの問題に悩まされた (毎日コードブックを配布)。

#### 非対称暗号が解決すること

1976年のディフィー・ヘルマン、1977年の RSA で:
- 公開鍵は公開して OK → 鍵配布問題が解決
- 秘密鍵は手元にあるだけ → 漏れない
- 公開鍵で暗号化 → 秘密鍵で復号
- 秘密鍵で署名 → 公開鍵で検証

代償:
- 計算が遅い
- 暗号化できるサイズが小さい (RSA-2048 で 245 バイト)
- 「公開鍵が本物か」を別途検証する必要 (→ PKI/CA)

### 解決策: ハイブリッド方式

```
1. 接続開始時、非対称暗号で「セッション鍵」を安全に共有 (遅いが少量)
2. 以降の本データは、セッション鍵で対称暗号化 (速い大量データ)
```

**「非対称は鍵を共有するためだけに使い、本通信は対称」**。

### TLS での具体的な使い分け

```
[ハンドシェイクフェーズ — 非対称暗号]
   ├ ECDHE: 共有秘密 S を確立 (鍵交換)
   ├ 証明書の署名検証: サーバー認証
   └ CertificateVerify: サーバーが秘密鍵を持つ証明

       ↓ ハンドシェイク完了

[本通信フェーズ — 対称暗号]
   ├ HTTP リクエスト/レスポンスを AES-256-GCM で暗号化
   └ MAC で完全性も保証 (GCM/Poly1305 は AEAD)
```

### 効率の差

1 GB の HTTP レスポンスをダウンロードする場合 (CPU 暗号処理時間):

| 方式 | 時間 |
|---|---|
| 全部 RSA | 約 8 分 |
| 全部 AES-256-GCM | 約 0.1 秒 |
| TLS ハイブリッド | 約 0.1 秒 + 数ミリ秒 (ハンドシェイク) |

### 比喩

> 非対称暗号 = 重くて精巧な金庫の鍵 (作るのも開けるのも時間がかかるが、安全に渡せる)
> 対称暗号 = 軽くて高速な鍵 (一瞬で開閉できるが、どう渡すかが問題)

非対称の金庫を使って軽い鍵を1回だけ送る → 以降は軽い鍵で開け閉め、というのが TLS の発想。

### TLS 1.2 と TLS 1.3 でハイブリッドの構造が違う

#### TLS 1.2 (RSA 鍵交換) — 古い
```
クライアントが premaster secret を公開鍵で暗号化して送る
→ 両者がセッション鍵を導出
```

#### TLS 1.2 (ECDHE) と TLS 1.3
```
両者が DH 公開鍵を交換
→ DH で共有秘密 S を計算 → セッション鍵を導出
```

「非対称で鍵を共有」の実現方法が、直接暗号化か DH 鍵交換かの違い。どちらも「少量データを非対称、本データは対称」の大枠は変わらない。

### 認証も非対称暗号の役割

「相手が本物か」も非対称暗号の仕事。これは暗号化ではなく**署名**:

```
サーバーが秘密鍵でハンドシェイク内容に署名 → CertificateVerify
クライアントが証明書の公開鍵で署名を検証
→ サーバーは証明書の正当な所有者
```

TLS 1.3 では非対称暗号の2つの用途のうち**署名がメイン**になっている。

### 整理表

```
                 [対称暗号]              [非対称暗号]
速度              超速い (GB/s)          超遅い (MB/s 以下)
鍵                1つ (両者で共有)       2つ (公開鍵+秘密鍵)
鍵配布            問題あり (どう渡す?)   問題なし (公開鍵は公開OK)
適する用途         大量データ            鍵交換、署名、認証
TLS での出番       本通信 (HTTP本文)     ハンドシェイク
```

### 使い分けの原則

- 「大量データを暗号化したい」→ 対称
- 「鍵を安全に共有したい / 相手の本人確認したい」→ 非対称
- 「実用的な暗号通信を作りたい」→ ハイブリッド (TLS 方式)

### 要点

- 対称は速いが鍵配布問題、非対称は鍵配布解決だが遅い
- 単独では実用にならず、組み合わせる (ハイブリッド方式)
- TLS のハンドシェイクは非対称、本通信は対称
- 非対称の用途は「鍵交換」と「署名 (認証)」の2つ
- TLS 1.3 では署名がメイン用途

## Q5: TLS のハンドシェイク・暗号化はどこで処理されている? NIC? カーネル? アプリ?

**基本的にはユーザー空間 (アプリ内の TLS ライブラリ)**。TCP はカーネルだが、TLS はカーネルではない。

### レイヤーごとの責務

```
[ HTTP (平文) ]                    ← アプリ内のロジック
       ↓
[ TLS (暗号化/復号) ]              ← ★ ユーザー空間の TLS ライブラリ
       ↓
[ TCP (信頼性転送) ]               ← カーネル
       ↓
[ IP (ルーティング) ]              ← カーネル
       ↓
[ NIC ドライバ ]                   ← カーネル
       ↓
[ NIC ハードウェア ]               ← 物理デバイス
```

カーネル/ユーザー空間の境界は **TCP と TLS の間**。TCP まではカーネルが面倒を見て、その上はアプリ任せ。

### NIC は TLS を知らない

NIC から見ると TLS の暗号文はただのバイト列。Ethernet/IP/TCP のヘッダだけ理解してペイロードはそのまま渡す。

### カーネルも TLS を知らない (通常は)

カーネルの TCP/IP スタックは TLS にノータッチ。TCP セグメントを順序通り並べて受信バッファに入れるだけ。「中身が平文か暗号化されたか」をカーネルは判別しない。**HTTP も HTTPS もカーネルから見ると区別がつかない** (違うのはポート番号くらい)。

### TLS ライブラリの代表例 (全部ユーザー空間)

| OS / 環境 | 代表的な TLS ライブラリ |
|---|---|
| Linux (汎用) | OpenSSL (圧倒的シェア) |
| Linux (Chromium 系) | BoringSSL (Google が fork) |
| Rust | rustls (Rust 製、メモリ安全) |
| Windows | SChannel (OS 組み込み) |
| Java | JSSE (Java 標準) |
| Go | crypto/tls (Go 標準、純 Go 実装) |
| Node.js | OpenSSL (バンドル) |

### HTTPS リクエスト1回の完全なトレース

#### 1. NIC がパケット受信
NIC が受信 → カーネルに割り込み → カーネルが IP/TCP ヘッダを解釈 → ソケット受信バッファへ。「TLS」「HTTP」は意識しない。

#### 2. アプリが read() で起きる
クライアントのアプリが `read()` syscall で寝ていた → 起きてバイト列を読み取る → 内部の TLS ライブラリに渡す。

#### 3. ユーザー空間で TLS 処理
TLS ライブラリ (OpenSSL/rustls など) がクライアントプロセス内で:
1. ServerHello を解釈
2. 証明書を取り出して CA で検証 (OS の信頼ストア参照)
3. ECDHE の鍵交換計算
4. セッション鍵を導出 (HKDF)
5. CertificateVerify の署名検証
6. Finished を計算

**全部プロセス内で計算**。カーネルからは「クライアントが何か計算してる」としか見えない。

#### 4. HTTP リクエスト送信 (暗号化)
```
クライアントのアプリ: GET / HTTP/2 を生成 (平文)
   ↓
TLS ライブラリ: AES-256-GCM で暗号化
   ↓ write() syscall
カーネル: TCP で送信
   ↓
NIC: 送信
```

#### 5. レスポンス受信 (復号)
逆方向。カーネルは暗号文と平文の違いを知らず、バイトを運ぶだけ。

### 図解

```
┌─ ユーザー空間 ─────────────────────────────────────┐
│  クライアント or サーバーのアプリ                  │
│  (curl, ブラウザ, nginx, axum, Rust プログラム等)  │
│    ├─ HTTP のロジック                              │
│    └─ TLS ライブラリ (OpenSSL / rustls など)       │
│         ├─ TLS ハンドシェイク                      │ ★ TLS はここ
│         ├─ 暗号化 (AES-256-GCM)                    │
│         └─ 復号 (AES-256-GCM)                      │
└─── system call ───────────────────────────────────┘
┌─ カーネル空間 ─────────────────────────────────────┐
│  TCP/IP スタック (ソケット、TCP、IP)               │
│  NIC ドライバ                                      │
└─── 割り込み ──────────────────────────────────────┘
┌─ ハードウェア ─────────────────────────────────────┐
│  NIC                                               │
└────────────────────────────────────────────────────┘
```

### なぜカーネルではなくユーザー空間か

設計の理由:

1. TLS はアプリケーション層の関心事 — 証明書検証ポリシー、暗号スイート選択はアプリごとに違う
2. 暗号アルゴリズムが進化する — TLS 1.0→1.2→1.3 の頻繁な更新。カーネル変更なしで済む
3. 脆弱性発見時の修正が速い — ライブラリ更新だけで済む
4. アプリごとに異なる TLS 実装を選べる — Rust なら rustls、Java なら JSSE

### 例外1: kTLS (Kernel TLS)

Linux 4.13 (2017) から **kTLS** という機能が入った。「ハンドシェイクはユーザー空間、本通信の暗号化/復号はカーネル」という妥協案。

- 利点: `sendfile()` syscall でユーザー空間コピー不要 → 高速
- Netflix が動画配信効率化のため主導開発
- 普及はまだ限定的 (nginx, Netflix 独自スタックなど)

### 例外2: ハードウェア支援

#### AES-NI (CPU の AES 専用命令)
Intel/AMD 現代 CPU の AES 専用命令。AES-256-GCM が 5〜10 GB/秒。OpenSSL や rustls が自動で利用。

#### NIC の TLS オフロード
Intel E810, NVIDIA ConnectX-7 などの NIC は TLS 暗号化をハードウェアで実行可能。Netflix など巨大配信で利用。

### TLS Termination — もう一つのパターン

アプリサーバーで TLS を処理せず、手前のロードバランサーで終端する。

```
クライアント ──HTTPS──→ [ロードバランサー/nginx] ──HTTP (平文)──→ アプリサーバー
              (TLS 終端)                          (内部ネット)
```

AWS ALB、GCP Cloud LB、Cloudflare などはこれ。利点:
- アプリサーバーで TLS 処理が不要 → 軽量
- 証明書管理を1箇所に集約
- ロードバランサーが HTTP の中身を見られる (ルーティング、キャッシュ)

「TLS は手前で終わらせ、内部はプレーン HTTP」が今のクラウドの主流。

### 処理の場所まとめ

| 処理 | 場所 | 担当 |
|---|---|---|
| 物理層フレーム送受信 | NIC | ハードウェア |
| NIC ドライバ | カーネル | ドライバ |
| IP ルーティング | カーネル | TCP/IP スタック |
| TCP 順序制御・再送 | カーネル | TCP/IP スタック |
| ソケットバッファ管理 | カーネル | TCP/IP スタック |
| **TLS ハンドシェイク** | **ユーザー空間** | **TLS ライブラリ** |
| **TLS 暗号化/復号** | **ユーザー空間** (kTLS なら一部カーネル) | **TLS ライブラリ + AES-NI** |
| **証明書検証** | **ユーザー空間** | **TLS ライブラリ + OS 信頼ストア** |
| HTTP リクエスト/レスポンス処理 | ユーザー空間 | アプリ |

### Rust から見ると

```rust
let response = reqwest::get("https://example.com/").await?;
```

この1行の裏で:
- reqwest が内部で rustls (または OpenSSL) を呼ぶ
- rustls がハンドシェイク・暗号化・復号を実行 (Rust コードとしてユーザー空間で動く)
- カーネルとは read/write syscall でやりとり
- カーネルは TCP セグメントを送受信するだけ
- NIC は物理層をやるだけ

### 要点

- TLS はカーネルではなく**ユーザー空間の TLS ライブラリ**で処理される
- NIC とカーネルは「暗号化か平文か」を知らず、ただバイトを運ぶ
- HTTPS リクエスト1回の流れ: NIC → カーネル (TCP) → アプリ (TLS 復号) → アプリ (HTTP 処理)
- 設計理由: TLS はアプリ層の関心事、進化が速い、アプリごとにカスタマイズしたい
- 例外: kTLS (カーネル TLS オフロード)、NIC ハードウェアオフロード
- 実運用では TLS Termination で手前のロードバランサーが TLS を終端、内部はプレーン HTTP もよくある
