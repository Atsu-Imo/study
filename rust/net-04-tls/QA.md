# net-04: TLS Q&A

日付: 2026-06-09

## Q1: なんで `署名^e mod n == digest` で digest が出るの？

RSA の核心。**`e` と `d` が互いに打ち消し合うように作ってあるから**。

### 1. 指数は掛け算でまとまる

署名は `signature = digest^d mod n`。これを `e` 乗すると累乗のルール `(x^a)^b = x^(a·b)` により:

```
signature^e = (digest^d)^e = digest^(d·e)   (mod n)
```

### 2. e と d は「φ(n) を法とした逆数」

鍵生成時に `e·d ≡ 1 (mod φ(n))` を満たすよう `d` を決めている (φ(n) は p,q から計算するオイラーのトーシェント)。「1 余る」= ある整数 k で:

```
d·e = 1 + k·φ(n)
```

これを指数に入れると:

```
digest^(d·e) = digest^(1 + k·φ(n)) = digest · (digest^φ(n))^k   (mod n)
```

### 3. オイラーの定理: digest^φ(n) ≡ 1 (mod n)

n と互いに素な数を φ(n) 乗すると必ず 1 (mod n) になる (フェルマーの小定理の一般化)。よって:

```
digest^(d·e) = digest · 1^k = digest   (mod n)
```

→ `signature^e mod n == digest` が成立する。

### このプロジェクトの数字で確認

`e=17, d=2753, φ(n)=3120, n=3233`:

```
d·e = 2753 × 17 = 46801 = 15 × 3120 + 1     ← φ(n) で割ると 1 余る
digest^46801 = (digest^3120)^15 × digest ≡ 1^15 × digest = digest  (mod 3233)
```

### 直感まとめ

- `d` 乗(署名)と `e` 乗(検証)は、指数を足すと `d·e` になる
- `d·e` は「φ(n) の倍数 + 1」に仕込んである
- φ(n) 乗ぶんはオイラーの定理で「何もしないのと同じ」
- 残った「+1」だけ効いて元の digest に戻る

`d` を知らない攻撃者は「e 乗すると digest に戻る数 (= 有効な署名)」を作れない (d か素因数分解が必要)。これが「CA だけが署名できる」根拠。

## Q2: client_random / server_random って何？公開鍵とは違う？

別物。**鍵ではなく、ただの使い捨て乱数 (ナンス)**。各自が生成して平文のまま送る。秘密でもなく盗聴OK。

| | DH 公開鍵 | client/server_random |
|---|---|---|
| 正体 | 秘密鍵から計算した値 (g^秘密 mod p) | ただのランダムな数 |
| 秘密鍵と関係 | ある (裏に秘密鍵がいる) | ない (何にも紐づかない) |
| 用途 | 共有秘密を作る材料 | セッション鍵に混ぜる「薬味」 |
| 公開/秘密 | 公開 | 公開 |

### 何のため？ → 毎回ちがうセッション鍵にする (鮮度 / freshness)

```
セッション鍵 = 導出( 共有秘密, client_random, server_random )
```

random なしで共有秘密だけから作ると、同じ DH 鍵を使い回したとき毎回同じセッション鍵になってしまい、昔の通信を投げ返す**リプレイ攻撃**が通る恐れがある。毎回ピチピチの乱数を混ぜることで、他の材料が同じでもセッション鍵は必ず毎回別物になる。

### たとえ

- 共有秘密 = 鍋のベースのスープ (DH で作る本体)
- random = 「今日の塩ひとつまみ」を両者が出し合って入れる (塩自体は秘密じゃないが、毎回ちがうので今日のスープは今日だけのものになる)
