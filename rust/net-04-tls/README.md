# net-04: TLS ハンドシェイク

## 概要

ネットワークの「登場人物」第4弾は **TLS** (Transport Layer Security) — HTTPS の "S"。

net-02 で作った HTTP は**平文のテキスト**だった。同じ Wi-Fi にいる人や経路上のルーターは、流れる `GET /login` も `password=...` も丸見えで、書き換えもできる。TLS は HTTP の**下**に潜り込んで、この通信を「盗聴できない・改ざんできない・なりすませない」状態にする層。ブラウザのアドレスバーの**鍵マークの正体**がこれ。

このプロジェクトでは本物の暗号ライブラリを使わず、**ハンドシェイクの「流れ」と「役者の受け渡し」を `std` だけで自作**して、

```
ClientHello → ServerHello → Certificate → 鍵交換 → 暗号化通信
```

の一連を TCP 上で実際に動かす。

> ⚠️ ここで使う鍵長・乱数・暗号はすべて教育用の **toy 実装**。本番では絶対に使わない（本番は鍵交換が数千ビット、暗号は AES-GCM、乱数は OS の CSPRNG）。目的は「数学の正しさ」ではなく「**誰が何を渡し、なぜそれで安全になるのか**」の理解。

## net-01〜03 との違い

| | net-02 (HTTP) | net-03 (DNS) | **net-04 (TLS)** |
|---|---|---|---|
| 中身 | 平文テキスト | バイナリ | **暗号文** |
| 守るもの | — | — | **機密性・完全性・真正性** |
| 鍵 | なし | なし | **公開鍵 + 共通鍵の合わせ技** |
| 登場人物 | サーバー | サーバー | **クライアント + サーバー + CA** |
| 核心 | リクエスト/レスポンス | バイト列パース | **ハンドシェイク** |

TLS は HTTP/DNS のような「1つのメッセージ形式」ではなく、**接続の最初に行う交渉（ハンドシェイク）の手順**そのものが主役。

## TLS が解決する3つの問題

| 問題 | 攻撃 | TLS の道具 |
|---|---|---|
| **機密性** (盗聴されたくない) | 経路上で覗き見 | 共通鍵暗号でデータを暗号化 |
| **完全性** (改ざんされたくない) | 途中でデータ書換え | メッセージ認証 (本物は MAC/AEAD) |
| **真正性** (なりすまされたくない) | 偽サーバーに接続させる | **証明書 + CA 署名** |

「暗号化」だけでは不十分なのがポイント。暗号化しても、**つなぎ先が偽サーバーなら**意味がない（偽サーバーと暗号通信してしまう）。だから「相手が本物か」を確かめる証明書が必須。

## ハンドシェイクの流れ

```
クライアント                                      サーバー
    │                                               │
    │ ① ClientHello (client_random, 使える暗号一覧)  │
    │ ─────────────────────────────────────────────>│
    │                                               │
    │ ② ServerHello (server_random, 採用する暗号)    │
    │ <─────────────────────────────────────────────│
    │ ③ Certificate (名前 + 公開鍵 + CA署名)         │
    │ <─────────────────────────────────────────────│
    │ ④ ServerHelloDone                             │
    │ <─────────────────────────────────────────────│
    │                                               │
    │  ★証明書を検証 (CA署名は正しい? 名前は一致?)    │
    │                                               │
    │ ⑤ ClientKeyExchange (クライアントの鍵材料)     │
    │ ─────────────────────────────────────────────>│
    │                                               │
    │  ★両者がそれぞれ「共有秘密」を計算 (= 同じ値)  │
    │  ★共有秘密 + 2つの random → セッション鍵を導出 │
    │                                               │
    │ ⑥ Finished (ここから暗号化) ⇄                  │
    │ <────────────────────────────────────────────>│
    │ ⑦ 暗号化された HTTP リクエスト/レスポンス ⇄    │
    │ <────────────────────────────────────────────>│
```

ポイントは **③④で「相手が本物か」を確かめてから、⑤⑥で「鍵を作る」** という順番。先に相手を信頼できると確認するからこそ、その後の鍵交換が意味を持つ。

## 登場人物の持ち物と、いつ何が流れるか（公開 / 秘密 / CA 早見表）

「どれが公開でどれが秘密で、どれが CA のもの？」がこんがらがりやすいので整理する。

記号: 🌐公開（誰が見てもいい・ネットに流れる）／🔒秘密（持ち主だけ・**絶対に流さない**）／🏛️CA（認証局がらみ）／🔑導出（流れないが両者が計算で手に入れる）／🔐暗号文（暗号化済み）

### (1) 誰が何を持っているか

| 持ち主 | 持ち物 | 区分 | ネットに流れる? | 役割 |
|---|---|---|:---:|---|
| 🏛️ CA | 署名用の秘密鍵 (d) | 🔒秘密 | ❌ | 証明書に署名する。**漏れたら全証明書を偽造され放題** |
| 🏛️ CA | 検証用の公開鍵 (e, n) | 🌐公開 | (事前配布) | **ブラウザ/OS に同梱**。信頼の起点 |
| サーバー | DH 秘密鍵 | 🔒秘密 | ❌ | サーバーだけが知る。共有秘密の計算に使う |
| サーバー | DH 公開鍵 | 🌐公開 | ✅(証明書の中) | 鍵交換の材料 |
| サーバー | 証明書 (名前+DH公開鍵+CA署名) | 🌐公開 | ✅ | 「自分は本物」の証明書 |
| サーバー | server_random | 🌐公開 | ✅ | セッション鍵を毎回変えるための乱数 |
| クライアント | DH 秘密鍵 | 🔒秘密 | ❌ | クライアントだけが知る |
| クライアント | DH 公開鍵 | 🌐公開 | ✅ | 鍵交換の材料 |
| クライアント | client_random | 🌐公開 | ✅ | 同上 |
| 両者 | 共有秘密 | 🔑導出 | ❌ | 各自が「相手の公開鍵 + 自分の秘密鍵」で計算。**同じ値になる** |
| 両者 | セッション鍵 | 🔑導出 | ❌ | 共有秘密 + 両 random から導出。実データの暗号化に使う |

### (2) いつ・どっちから・何が送られるか（時系列）

| 順 | メッセージ | 方向 | 中身 | 区分 |
|:---:|---|:---:|---|---|
| ① | ClientHello | C → S | client_random, 使える暗号の一覧 | 🌐 全部公開 |
| ② | ServerHello | S → C | server_random, 採用する暗号 | 🌐 全部公開 |
| ③ | Certificate | S → C | 名前 + サーバー DH 公開鍵 + **CA 署名** | 🌐公開（🏛️署名入り） |
| ④ | ServerHelloDone | S → C | （空。「送るものは送った」の合図） | — |
| ★ | 〔クライアントが証明書を検証〕 | C 内部 | CA 公開鍵で署名をチェック・名前を照合 | 🏛️ |
| ⑤ | ClientKeyExchange | C → S | クライアント DH 公開鍵 | 🌐 公開 |
| ★ | 〔両者が共有秘密 → セッション鍵を計算〕 | 各自 | 計算するだけ・**何も流れない** | 🔑 |
| ⑥ | Finished ⇄ | C ⇄ S | 暗号化した「準備OK」確認 | 🔐 暗号文 |
| ⑦ | Application Data ⇄ | C ⇄ S | 暗号化した HTTP リクエスト/レスポンス | 🔐 暗号文 |

### (3) 一番大事な気づき

- **①〜⑤でネットに流れるものは全部「公開」**。盗聴者には丸見え。それでいい。
- **秘密鍵（CA・サーバー・クライアントの3種）は、一度もネットに出ない。**
- **共有秘密もセッション鍵もネットに出ない** — 各自が手元で計算して同じ値にたどり着くだけ。
- だから盗聴者は「流れる公開情報を全部集めても」セッション鍵を作れない → ⑥⑦の中身を読めない。
- ③を攻撃者がすり替えても、攻撃者は **CA 秘密鍵を持たない＝有効な署名を作れない**ので、★の検証でバレる。
- 「公開鍵」は**3組**ある（CA・サーバー・クライアント）。混同しやすいので、
  「CA の鍵＝**署名/検証**用」「サーバー/クライアントの DH 鍵＝**鍵共有**用」と用途で分けて覚えるとよい。

## 核心1: なぜ公開鍵暗号と共通鍵暗号を両方使うのか

これが TLS で一番大事な考え方。

| | 共通鍵暗号 (対称) | 公開鍵暗号 (非対称) |
|---|---|---|
| 鍵 | 1個を双方で共有 | 公開鍵 + 秘密鍵のペア |
| 速度 | **速い** | **遅い** (数百〜千倍) |
| 課題 | **どうやって鍵を安全に渡す?** | 速度 |
| 例 | AES, ChaCha20 | RSA, ECDH, Diffie-Hellman |

- 共通鍵暗号は速いが、「最初の1個の鍵をどうやって相手に渡すか」が問題（送る途中で盗聴されたら終わり）。
- 公開鍵暗号は「鍵を安全に共有する」のが得意だが、遅い。

→ **答え: いいとこ取り。**
**最初の鍵共有だけ公開鍵暗号（遅くてもOK、一瞬だから）→ 共有できた鍵で以降のデータを共通鍵暗号（速い）で暗号化。**

このプロジェクトでは鍵共有に **Diffie-Hellman (DH)**、データ暗号化に **XOR ストリーム暗号 (toy)** を使っている。

### Diffie-Hellman 鍵交換の仕組み

公開情報（素数 `p`、生成元 `g`）を共有した上で:

```
クライアント                            サーバー
秘密鍵 a を決める                       秘密鍵 b を決める
公開鍵 A = g^a mod p                    公開鍵 B = g^b mod p
        │  A を送る  ──────────────────>│
        │<──────────────  B を送る      │
共有秘密 = B^a mod p             共有秘密 = A^b mod p
        = (g^b)^a = g^(ab) mod p = (g^a)^b = 同じ値！
```

魔法のような点: **秘密鍵 `a`, `b` は一度もネットワークに流れない**。流れるのは `A`, `B` (公開鍵) だけ。それでも両者は同じ `g^(ab) mod p` にたどり着ける。

盗聴者は `p, g, A, B` を全部見られるが、`A = g^a mod p` から `a` を逆算する（**離散対数問題**）のが、`p` が十分大きいと現実的時間で解けない。これが安全性の根拠。（デモの最後 `eavesdropper_demo()` でこの様子を出力する。）

## 核心2: 証明書と CA — 「相手が本物か」の確認

DH 鍵交換だけだと**中間者攻撃 (MITM)** に弱い。攻撃者が間に入って、クライアントとは攻撃者の鍵で、サーバーとは別の鍵で、2本のDHを張れてしまう（両側と暗号通信できてしまう）。

これを防ぐのが **証明書**。サーバーは「自分の名前 + 公開鍵」に **CA (認証局) の署名**が付いたものを提示する。

```
証明書 = { name: "toytls.local", 公開鍵: B, 署名: CA秘密鍵で署名した値 }
```

- **CA の公開鍵はブラウザ/OS に最初から同梱**されている（信頼の起点 = トラストアンカー）。
- クライアントは CA の公開鍵で署名を検証 → 「確かに CA が、この名前とこの公開鍵の組を保証した」と分かる。
- 攻撃者は **CA の秘密鍵を持っていない**ので、偽の証明書に有効な署名を付けられない → MITM は検出される。

このプロジェクトでは toy RSA で署名/検証を実装している:

- 署名 (CA): `署名 = digest^d mod n` （`d` = CA 秘密鍵）
- 検証 (client): `署名^e mod n == digest` か確認 （`e, n` = CA 公開鍵）

`digest` は「名前 + サーバー公開鍵」のハッシュ。だから名前や鍵を1ビットでも改ざんすると digest が変わり、検証に失敗する（演習1で体験）。

## Go との比較

Go で TLS クライアントを書くなら標準ライブラリの `crypto/tls`:

```go
conn, err := tls.Dial("tcp", "example.com:443", &tls.Config{
    ServerName: "example.com", // 証明書の名前検証に使う
})
// ハンドシェイク・証明書検証・鍵交換・暗号化は全部ランタイム任せ
conn.Write([]byte("GET / HTTP/1.1\r\nHost: example.com\r\n\r\n"))
```

これで終わり。証明書チェーンの検証もシステムのルート CA ストアを使って自動。

Rust でも本番は `rustls` クレートを使えば同レベル:

```rust
// rustls を使う場合 (本番)。今回はこれを使わず std で中身を自作。
let config = rustls::ClientConfig::builder()
    .with_root_certificates(root_store)
    .with_no_client_auth();
```

今回は**あえて手書き**で、ライブラリが裏で何をやっているかを暴く。

| 概念 | Go | Rust (今回の toy) |
|---|---|---|
| 接続 | `tls.Dial` | `TcpStream::connect` + 自作ハンドシェイク |
| 証明書検証 | ランタイム自動 | `verify_cert` (RSA 署名検証) |
| 鍵交換 | ランタイム自動 | `mod_pow` による DH |
| 暗号化 | ランタイム自動 | `xor_cipher` (toy) |
| ルート CA | システムストア | `RSA_E, RSA_N` 定数 |

## コード解説

### 鍵交換の心臓部: 冪剰余

```rust
fn mod_pow(base: u64, exp: u64, modulus: u64) -> u64 {
    let m = modulus as u128;          // 溢れ防止に u128
    let mut result: u128 = 1;
    let mut b = (base as u128) % m;
    let mut e = exp;
    while e > 0 {
        if e & 1 == 1 { result = result * b % m; }
        e >>= 1;
        b = b * b % m;
    }
    result as u64
}
```

DH の `g^a mod p` も、RSA の `m^d mod n` も、すべてこれ。公開鍵暗号は「冪剰余は速く計算できるが、逆（離散対数・素因数分解）は難しい」という非対称性の上に成り立っている。

### 証明書の署名と検証

```rust
fn ca_sign(name: &str, dh_public: u64) -> u64 {     // CA だけが d を持つ
    mod_pow(cert_digest(name, dh_public), RSA_D, RSA_N)
}
fn verify_cert(name: &str, dh_public: u64, sig: u64) -> bool {  // 誰でも e で検証
    mod_pow(sig, RSA_E, RSA_N) == cert_digest(name, dh_public)
}
```

### レコード層 (フレーミング)

TLS は1本の TCP ストリームの上に複数のメッセージを流すので、区切りが要る。`[種別1B][長さ4B][本体]` で枠を付けている（本物の TLS レコードの簡略版）。

## 実装上の注意

### これは「おもちゃ」である

- DH の鍵長が 61bit、RSA が 12bit 相当 — 本番なら一瞬で破られる。安全性は出ない。
- 乱数が `SystemTime` ベース — **予測可能**で暗号には致命的。本番は OS の CSPRNG。
- XOR ストリーム暗号 + MAC なし — 改ざん検知ができない。本番は AES-GCM 等の AEAD。

学ぶのは「**手順と役割**」であって「この実装の安全性」ではない。

### 本物との対応・簡略化した点

| 本物の TLS | 今回の toy |
|---|---|
| 証明書チェーン (ルート→中間→サーバー) | CA 直署名の1枚（チェーンは演習3） |
| ephemeral DH (毎回使い捨て) で前方秘匿性 | サーバー鍵は固定（CA が署名できるように） |
| 鍵導出は HKDF/PRF | toy な XOR + 回転 + LCG |
| TLS 1.3 は 1-RTT・暗号スイート交渉が簡略化 | TLS 1.2 風の往復に近い形 |

### 前方秘匿性 (Forward Secrecy)

今回はサーバー DH 鍵を固定したが、本物（TLS 1.3）は**毎回使い捨ての DH 鍵 (ephemeral)** を使う。こうすると、後でサーバーの長期秘密鍵が漏れても、過去に記録された通信は復号できない。クライアント鍵だけ ephemeral にしているのはこの予習。

## まとめ

- TLS は **HTTP の下に潜って通信を暗号化する層**。HTTPS の "S"。
- 守るのは **機密性・完全性・真正性** の3つ。暗号化だけでなく「相手が本物か」の確認が要る。
- ハンドシェイク順は **Hello で交渉 → 証明書で相手確認 → 鍵交換 → 暗号化通信**。
- **公開鍵暗号で鍵共有（遅いが安全）→ 共通鍵暗号でデータ（速い）** の合わせ技がTLSの肝。
- **Diffie-Hellman** は秘密鍵を流さずに共有秘密を作る。安全性の根拠は離散対数問題。
- **証明書 + CA** がなりすまし（MITM）を防ぐ。CA の署名を CA の公開鍵で検証する。
- ブラウザが信頼の起点として持つのは **ルート CA の公開鍵**だけ。
- 次は **net-05 (CA)** で、この「証明書の発行・署名・信頼の連鎖」を主役として深掘りする。
