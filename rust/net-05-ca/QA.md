# net-05: 認証局 (CA) Q&A

日付: 2026-06-09

## Q1: (openssl s_client の出力を見て) 署名されたものが正当だとどう判断している？

本物のチェーン (example.com) も net-05 と同じ構造。`s:`=Subject(持ち主)、`i:`=Issuer(発行者)で、下の段の `i:` が1つ上の段の `s:` と一致してバケツリレーになっている。

```
depth 0  s: example.com                       i: Cloudflare TLS Issuing ECC CA 3
depth 1  s: Cloudflare TLS Issuing ECC CA 3   i: SSL.com TLS Transit ECC CA R2
depth 2  s: SSL.com TLS Transit ECC CA R2     i: SSL.com TLS ECC Root CA 2022
depth 3  s: SSL.com TLS ECC Root CA 2022      i: AAA Certificate Services (クロス署名)
```

### 判断の2本の柱

**柱1: 署名の検証 — その鍵でしか作れない**
各証明書の署名を「1つ上の段の公開鍵」で検証する (openssl の `verify return:1` が各段OKの印)。
→ **公開鍵で検証が成功する署名は、対応する秘密鍵を持つ者にしか作れない**。秘密鍵は発行者だけが持つので、「検証が通った=確かにその発行者が、この中身そのままに署名した」と断定できる。1ビット改ざんで失敗 (シナリオB)。
※ 本物は ECDSA、net-05 は RSA。数式は違うが「秘密鍵で署名・公開鍵で検証」の役割は同じ。

**柱2: トラストアンカーに到達するか — 自分が元から信じる根に繋がるか**
署名が正しくても、チェーン上端が「自分のPC/ブラウザに元から入っているルートCA」に繋がらないと無意味。ルート公開鍵はサーバーから送られず、OS/ブラウザのルートストアにある。たどり着けたら `Verification: OK`。
→ 攻撃者が偽チェーンを作っても、手元のストアの本物ルート鍵に署名で繋げられないので弾かれる (シナリオD)。**信頼は「名前」ではなく「自分の信じる鍵に署名で繋がるか」で決まる**。

### 署名以外のチェック
- 有効期限 NotBefore〜NotAfter (例: example.com は 90日運用)（シナリオC）
- ホスト名の一致 = **SAN (Subject Alternative Name)** に一致名があるか (現代は CN でなく SAN)
- CAとして署名してよいか (BasicConstraints CA:TRUE)、失効していないか (CRL/OCSP)

### クロス署名
depth3 の SSL.com ルートは、さらに上が AAA Certificate Services になっている。新ルートを既存の広く信頼された古ルートにも署名してもらい、手元に新ルートが無くても古ルート経由で辿れるようにする保険。

### ひとこと
「1つ上の公開鍵で署名が検証でき」、それを繰り返して「自分のPCが元から信頼するルートまで途切れず辿れた」とき正当と判断する。署名は秘密鍵を持つ者にしか作れず、ルート鍵は自分の手元にしかないので攻撃者は割り込めない。

(なお出力の `Negotiated TLS1.3 group: X25519MLKEM768` は証明書検証ではなく net-04 の鍵交換=DHの部分。証明書検証=「相手が本物か」、鍵交換=「秘密の共有」と役割が別。)

## Q2: 証明書の検証って普段実装しない。フレームワークが勝手にやってくれる認識でいい？

おおむね正解。HTTPS 通信するライブラリはデフォルトで検証を全部やる (Go net/http, Rust reqwest, Node axios, Python requests, curl, gRPC, DBドライバの TLS なども)。信頼するルートCAも OS のルートストアを自動利用。自分で verify_chain を書くことは普段ない。net-05 は中身を知るための解剖。

### 実務で“触る”場面
- 社内/オレオレ証明書・企業プロキシでエラー → **そのCAをトラストストアに足す**
- 期限切れ・ホスト名不一致 → 証明書を直す/更新する
- mTLS (相互認証) → 自分のクライアント証明書を設定
- 証明書ピンニング (モバイル等) → あえて検証を上乗せ

### 一番大事な落とし穴
エラーを「検証を無効化して黙らせる」のが世界一やりがちな事故:
- Go `tls.Config{InsecureSkipVerify: true}`
- `curl -k` / Python `requests.get(url, verify=False)`

これをやると暗号化はされても「相手が本物か」を確認しないので**中間者攻撃に丸腰**。正しい対処は「検証を切る」ではなく「信頼するCAを足す / 証明書を直す」。

→ ひとこと: **普段はフレームワーク任せでOK。ただしエラーが出ても検証を切らない。CAを足すか証明書を直す。** net-05 を学ぶ実務的意味は「なぜ検証を切ってはいけないか」を腹落ちさせること。

## Q3: オレオレ証明書って「自分でルートCAを立てて、階層がない証明書を返す」で合ってる？

惜しい。実は2つの別パターンが混ざっている。

**パターン1: 自己署名証明書 (一番素朴なオレオレ)**
- 1枚の証明書が自分で自分に署名。Subject == Issuer。**階層ゼロ**。
- net-05 の root_cert ("Root CA"→"Root CA") がこれ。
- → 「階層がない証明書」はこっち。

**パターン2: 自前ルートCAを立てて発行 (プライベートCA)**
- 自分のルートCAでサーバー証明書に署名。**階層はある** (自前ルート→サーバー)。
- 本物のPKIと同じ構造。違いは「ルートが公的に信頼されていない」だけ。
- → 「自分でルートCAを立てて」はこっち。

**共通する本質 (階層の有無は本質ではない):**
> チェーンの根(ルート)が、相手のトラストストアに入っていない。
だからブラウザは「知らないCAだ」と警告する。net-05 シナリオD（知らない根は弾かれる）と同じ理屈。

**使い分け:**
| | 自己署名(1枚) | 自前ルートCA |
|---|---|---|
| 階層 | なし | あり |
| 信頼のさせ方 | その証明書をストアに足す | ルートCAだけ足せば配下は全部OK |
| 用途 | 使い捨て/ローカル | 社内の複数サーバー運用 |

社内で複数台運用なら、1枚ずつ信頼させるより自前ルートCAを1回信頼させる方が楽。だから「ちゃんとしたオレオレ」は普通パターン2を指す。解決は検証OFFではなく「その根/証明書を信頼ストアに足す」。

## Q4: CAサーバーは具体的にどう動く？ どのタイミングで証明書をクライアントとやりとりする？

大前提: **CAはTLS接続中(サイトを開く瞬間)には登場しない**。時間軸が2つある。

### 時間軸A: 発行 (事前・数ヶ月に1回) … サーバー運営者 ⇄ CA
```
① 鍵ペア生成 (秘密鍵はサーバー保管、CAには渡さない)
② CSR作成 = 公開鍵+ドメイン名 を自分の秘密鍵で署名 ──> CA受領
③ CA <── ドメイン所有の確認(チャレンジ): HTTP-01 (.well-known/acme-challenge/にファイル) / DNS-01 (TXTレコード)
④ 設置して応答 ──> CA「確かに example.com を支配」と確認
⑤ CA <── 署名済み証明書 + 中間CA証明書 を発行 (中身は net-05 の issue_cert そのもの)
⑥ サーバーに設置、以後使い回す
```
- CAの仕事は「ドメイン所有確認 → 署名」だけ。秘密鍵はCAに渡らない(見るのは公開鍵=CSRだけ)。
- Let's Encrypt/certbot はこれを **ACME** プロトコルで自動化。「CAサーバー」=ACMEサーバー+署名インフラ。
- 署名鍵は **HSM(耐タンパーHW)**。ルート鍵は普段オフラインの金庫。

### 時間軸B: 接続 (アクセス毎) … ブラウザ ⇄ Webサーバー (CA不在)
証明書をクライアントに渡すのは **TLSハンドシェイクの Certificate メッセージ (net-04 の③)**。
**送り主はWebサーバー**(CAではない)。事前にCAから貰った[サーバー証明書+中間CA]を接続のたびに提示する。
openssl s_client で見た Certificate chain がこの瞬間に届いたもの。

### 一番大事: 接続中ブラウザはCAに聞かない
- 検証は**手元のルートストアだけで完結** (だから速い・CAが落ちててもサイトは開く)。
- 例外は**失効確認(OCSP/CRL)**のみ。
- ただし今は **OCSPステープリング**が主流: サーバーが事前にCAから「まだ有効」の署名証明を貰っておき、ハンドシェイク時に同梱。ブラウザはCAに接続不要。

### まとめ
CAサーバー=「事前の発行(ドメイン確認+署名)」専門。証明書をクライアントに渡すのはTLSハンドシェイクの Certificate メッセージで送り主はWebサーバー。ブラウザの検証は手元のストアで完結、CAに聞くのは失効確認くらい(それもステープリングで肩代わり)。
