# net-05: 認証局 (CA) と信頼の連鎖

## 概要

ネットワークの「登場人物」第5弾は **CA (Certificate Authority / 認証局)** — 「この証明書は本物ですよ」と保証するハンコ屋。

net-04 (TLS) で、サーバーは「自分の名前 + 公開鍵」に **CA の署名**が付いた証明書を提示した。あれで「なりすまし」を防いでいた。net-05 ではその **証明書そのもの**にフォーカスして、

```
ルートCA  ──署名──→  中間CA  ──署名──→  サーバー証明書
```

という多段の **「信頼の連鎖 (chain of trust)」** を実際に組み立てて検証する。

> ⚠️ net-04 と同じく暗号は教育用 toy（鍵が極小）。学ぶのは「証明書の発行・署名・検証がどういう役割で繋がっているか」であって、実装の安全性ではない。本番は OpenSSL / rustls + 本物の X.509。

## net-04 との違い

| | net-04 (TLS) | **net-05 (CA)** |
|---|---|---|
| 主役 | ハンドシェイクの流れ | **証明書と信頼の連鎖** |
| CA | 1段（CAが直接サーバーに署名） | **多段（ルート→中間→サーバー）** |
| 検証 | 「署名が正しいか」1回 | **チェーンを根元までたどる** |
| 通信 | あり（TCP） | **なし**（証明書のロジックだけ） |

net-04 の演習3「信頼の連鎖を実装しよう」を、本編として展開した回。

## CA とは — 「公開鍵の持ち主」を保証する第三者

公開鍵暗号には根本的な問題がある:

> 「この公開鍵は、本当に `toytls.local` のもの？ 攻撃者がすり替えた偽物では？」

公開鍵そのものには名前が書いていない。ただの数字。だから**「この公開鍵は確かにこの名前の持ち主のものだ」と、信頼できる第三者に保証してもらう**仕組みが要る。それが CA。

```
あなた「toytls.local の公開鍵はこれで合ってる？」
CA   「うん、その公開鍵は確かに toytls.local のものだと私が保証する（署名）」
```

CA が「名前 + 公開鍵」のセットに署名したもの = **証明書**。

## 証明書の中身

実物の X.509 証明書はもっと項目が多いが、本質はこれだけ:

| 項目 | 意味 | このプロジェクト |
|---|---|---|
| **Subject** | 持ち主の名前 | `toytls.local` |
| **Issuer** | 発行者（署名した人）の名前 | `Intermediate CA` |
| **NotAfter** | 有効期限 | `20270609` |
| **PublicKey** | 持ち主の公開鍵 | `(e, n)` |
| **Signature** | Issuer の秘密鍵による署名 | 数値 |

ポイント: **Subject（持ち主）と Issuer（発行者）は別人**。サーバー証明書の持ち主は `toytls.local` だが、署名したのは `Intermediate CA`。この「持ち主 ≠ 発行者」が連鎖を生む。

`openssl s_client -connect example.com:443` の出力に出てくる `subject=` `issuer=` がまさにこれ。

## 信頼の連鎖 (Chain of Trust)

```
        ┌─────────────────────────────┐
        │ ルートCA証明書              │
        │ Subject = Root CA           │  ← 自己署名 (Subject == Issuer)
        │ Issuer  = Root CA           │  ← ブラウザ/OS に同梱（信頼の起点）
        └──────────────┬──────────────┘
                       │ ルートCA の秘密鍵で署名
                       ▼
        ┌─────────────────────────────┐
        │ 中間CA証明書                │
        │ Subject = Intermediate CA   │
        │ Issuer  = Root CA           │
        └──────────────┬──────────────┘
                       │ 中間CA の秘密鍵で署名
                       ▼
        ┌─────────────────────────────┐
        │ サーバー証明書              │
        │ Subject = toytls.local      │
        │ Issuer  = Intermediate CA   │
        └─────────────────────────────┘
```

### 検証は「下から上へ根元をたどる」

クライアントが最初から信頼しているのは **ルートCAの公開鍵だけ**（トラストアンカー）。サーバーは `[サーバー証明書, 中間CA証明書]` の2枚を送る（**ルートは送らない** — クライアントが既に持っているから）。

1. **中間CA証明書**を **ルートCAの公開鍵**で検証 → 中間CAはルートが保証した本物か？
2. **サーバー証明書**を **中間CAの公開鍵**（=①で検証済みの証明書に載っている）で検証 → サーバーは中間CAが保証した本物か？
3. 各証明書の**有効期限**を確認
4. サーバー証明書の **Subject** がアクセス先ホスト名と一致するか確認

「①で検証した証明書の中の公開鍵を使って②を検証する」という**バケツリレー**になっているのがミソ。根元（ルート）まで遡れて初めて信頼できる。

### なぜ中間CAをはさむのか

「ルートCAが直接サーバーに署名すればよくない？」と思うが、実務では中間CAを使う:

- **ルートCAの秘密鍵を金庫にしまえる**: ルート鍵は最重要（漏れたら全部偽造される）。普段はオフラインの金庫に保管し、日々の発行は中間CAにやらせる。
- **被害を限定できる**: 中間CAの鍵が漏れても、その中間CAだけ失効させればよく、ルートは無事。
- **ルート証明書は超長期**（20〜30年）。配布済みで更新が難しいので、めったに使わない。

## Go との比較

Go で証明書チェーンを検証するなら `crypto/x509`:

```go
roots := x509.NewCertPool()       // 信頼するルートCAの集合（トラストアンカー）
roots.AddCert(rootCert)

opts := x509.VerifyOptions{
    DNSName:       "toytls.local", // 名前の一致チェック
    Intermediates: intermediates,  // サーバーから受け取った中間CA
    Roots:         roots,
}
_, err := serverCert.Verify(opts)  // チェーンをたどって検証 → これ1発
```

`Verify` 1発の裏で、今回手書きした「チェーンを根元までたどる・署名検証・期限・名前」を全部やってくれている。

| 概念 | Go | Rust (今回の toy) |
|---|---|---|
| トラストアンカー | `x509.CertPool` | `trust_anchor: (&str, (u64,u64))` |
| 証明書 | `x509.Certificate` | `struct Certificate` |
| 発行（署名） | `x509.CreateCertificate` | `issue_cert` |
| チェーン検証 | `cert.Verify(opts)` | `verify_chain` |
| 名前検証 | `VerifyOptions.DNSName` | `server.subject != hostname` |

## コード解説

### 証明書の発行 = 署名対象を作って署名する

```rust
fn issue_cert(subject, issuer_name, not_after, subject_pubkey, issuer_key) -> Certificate {
    let tbs = cert_tbs(subject, issuer_name, not_after, subject_pubkey); // 署名対象
    let digest = toy_hash(&tbs) % issuer_key.n;        // 中身を1つの数値に潰す
    let signature = issuer_key.sign(digest);           // issuer の秘密鍵で署名
    Certificate { subject, issuer, not_after, public_key, signature }
}
```

**TBS (To Be Signed)** = signature 以外の全項目。これをハッシュして署名する。だから**どの項目を1ビット変えても署名が合わなくなる**（改ざん検知）。

### 検証は署名の逆操作

```rust
fn verify_signature(cert, issuer_pubkey) -> bool {
    let tbs = cert_tbs(&cert.subject, &cert.issuer, cert.not_after, cert.public_key);
    let digest = toy_hash(&tbs) % n;
    mod_pow(cert.signature, e, n) == digest   // s^e mod n が digest に戻るか
}
```

「`署名^e mod n` が元の digest に戻る」のが RSA の核心（→ net-04 の QA.md Q1 参照）。**数式は忘れてよくて**、「発行者の秘密鍵でしか作れない署名を、発行者の公開鍵で誰でも検証できる」という役割だけ掴めば十分。

## 動作確認

```bash
cd net-05-ca
cargo run
```

4つのシナリオが流れる:

| シナリオ | 内容 | 結果 |
|---|---|---|
| **A** | 正規のチェーン | ✅ OK |
| **B** | 攻撃者が期限を勝手に延長（改ざん） | ❌ 署名NG（中身を変えると digest が変わる） |
| **C** | 期限切れの証明書 | ❌ 期限NG |
| **D** | 攻撃者が自前のCAで発行（名前だけ "Root CA" を騙る） | ❌ 署名NG（鍵が違う） |

特に **D が重要**: 攻撃者は名前を `Root CA` と偽れても、**本物のルート秘密鍵を持っていない**ので有効な署名を作れない。→ **「信頼は名前ではなく『鍵による署名』で成り立つ」**。

### 本物を覗く

```bash
# 証明書チェーンが Subject/Issuer 付きで表示される
openssl s_client -connect example.com:443 -showcerts

# 単一の証明書の中身を読む
openssl x509 -in cert.pem -text -noout
```

出力の `subject=` `issuer=` を上から下にたどると、今回作ったチェーンと同じ構造（サーバー → 中間CA → ルート）が見える。

## 実装上の注意（toy であること）

- 鍵が極小（n が数千）なので、一瞬で素因数分解されて秘密鍵がバレる。安全性はゼロ。
- ハッシュが toy（衝突しやすい）。本物は SHA-256。
- 日付を `YYYYMMDD` の整数で比較しているだけ。本物は ASN.1 の時刻型。
- 本物の X.509 は他にも多数の項目（シリアル番号、拡張領域＝SAN/KeyUsage/BasicConstraints など）を持つ。特に **SAN (Subject Alternative Name)** が現代のホスト名検証の主役（CN ではなく）。

## 振り返りチェックリスト

- [ ] **自己署名証明書**とは何か（Subject == Issuer、ルートCAがこれ）
- [ ] **信頼の連鎖**を「ルート → 中間 → サーバー」の順で図に描ける
- [ ] 検証が「**下から根元（ルート）へバケツリレー**」で進むことを説明できる
- [ ] クライアントが最初から持つのは **ルートCAの公開鍵だけ**だと理解している
- [ ] サーバーが送るのは **サーバー証明書 + 中間CA証明書**（ルートは送らない）と言える
- [ ] **改ざん**するとなぜ署名検証に失敗するか（TBS の digest が変わる）を説明できる

- [ ] **中間CAをはさむ理由**（ルート鍵を金庫に・被害限定）を1つ挙げられる

## まとめ

- **CA** は「この公開鍵はこの名前の持ち主のもの」と保証する信頼できる第三者。
- **証明書** = 「名前 + 公開鍵」に発行者(Issuer)が署名したもの。Subject(持ち主) ≠ Issuer(発行者)。
- **信頼の連鎖**: ルートCA（自己署名）→ 中間CA → サーバー証明書、と署名でつながる。
- **検証**は下から根元へたどる。信頼の起点はブラウザ同梱の**ルートCA公開鍵だけ**。
- **改ざんは署名検証で**、**偽CAは「鍵が違う」ことで**弾かれる。信頼は名前ではなく鍵で成り立つ。
- **中間CA**をはさむのは、ルート鍵を厳重保管し被害を限定するため。
- 次は **net-06 (HTTPS Proxy)** で、中継者（フォワードプロキシ）と CONNECT トンネリングに進む。
