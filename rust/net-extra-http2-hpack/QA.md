# net-extra (番外編): HTTP/2 ヘッダー圧縮 (HPACK) と HTTP/2 Bomb Q&A

日付: 2026-06-05

## Q1: ヘッダーがバイナリでどうやって読まれているのか、いまいち分からない

HTTP/1.1 (net-02) は「改行区切りのテキスト」だったので `split_once(':')` で読めた。
HTTP/2 はそこが根本的に違い、ヘッダーは **HPACK (RFC 7541) というバイナリ** で送られる。

### 核心は「先頭ビットで表現形式を判定 → テーブルを引く」

例えば `:method: GET` はワイヤ上でたった1バイト `0x82`:

```
0x82  =  1000 0010
         │   └────── 下位7ビット = 0000010 = index 2 (= :method: GET)
         └────────── 先頭ビット 1 = "Indexed Header Field" (テーブル参照)
```

サーバーがやっているのは次の判定だけ:

| 先頭ビット | 表現形式 | 意味 |
|---|---|---|
| `1xxxxxxx` | Indexed (§6.1) | テーブルの index 番をまるごと参照 |
| `01xxxxxx` | Literal + 動的テーブル登録 (§6.2.1) | 新しいヘッダーを送り、表にも追加 |
| `0000xxxx` | Literal (登録なし) (§6.2.2) | 1回限りのヘッダー |

`src/main.rs` の `decode()` がまさにこれで、`b & 0x80`(参照か)、`b & 0x40`(新規登録か)
で分岐している。これが「ヘッダーをバイナリで読む」処理の本体。

### なぜテキストで読めないのか

1. **バイナリフレーミング** — HTTP/2 は全部「フレーム」というバイナリ単位で送る。
   1本の TCP に複数ストリームを多重化するため、テキストの行区切りでは表現できない。
2. **HPACK 圧縮** — ヘッダーは平文ではなく「テーブル参照」に圧縮される。
   静的テーブル61個 + 接続ごとに育つ動的テーブルを共有し、index で参照する。
   `:method: GET` (HTTP/1.1 ではテキスト11バイト) が HTTP/2 では1バイトになる。

### 数値は可変長整数 (§5.1)

index や長さは「先頭バイトの下位 N ビット」に詰め、入りきらなければ後続バイトに
7ビットずつ分割する (最上位ビット `0x80` = "まだ続く")。

```
index 5    → [0x85]              (1バイト)
index 62   → [0xbe]              (1バイト)   ← 127未満は全部1バイト
index 127  → [0xff, 0x00]        (2バイト)
index 1337 → [0xff, 0xba, 0x09]  (3バイト)
```

`cargo run -p net-extra-http2-hpack` のパート1〜2で、この実バイトを
16進・2進・ASCII で1バイトずつ確認できる。

## Q2: HTTP/2 Bomb (CVE-2026-49975) はこの仕組みとどう関係する?

HPACK の圧縮効率 (1バイトでヘッダー参照) が、そのまま増幅攻撃の燃料になる。

1. **HPACK 増幅**: 4KB の `cookie` を1回だけ送って動的テーブル index 62 に登録 →
   以降その index への **1バイト参照を連打** → サーバーは参照ごとに 4KB をメモリ復元。
   送信1バイト : 復元4,102バイト ≈ **4,000:1 の増幅** (デモのパート3で実測できる)。
2. **Slowloris 保持**: フロー制御ウィンドウを0にし、1バイトの `WINDOW_UPDATE` を
   送り続けてストリームを完了させない → 確保したメモリが解放されない。

```
増幅 (HPACK)  ×  保持 (Slowloris)  =  数十GBのメモリ枯渇 → サーバー停止
```

家庭用100Mbps・1台のPCから、Envoy/Apache に 32GB を10〜18秒で確保させられる。
詳細・パッチ状況・緩和策は README.md の「HTTP/2 Bomb」節を参照。

## Q3: 新しく登録されたヘッダーは、どの単位で動的テーブルに保存される? プロセスごと?

**プロセスごとではなく「コネクション(TCP接続)ごと」。** 粒度は「プロセス > 接続 > リクエスト」の真ん中。

```
サーバープロセス
├─ コネクションA (client1)  → 専用の動的テーブルA
├─ コネクションB (client2)  → 専用の動的テーブルB   ← Aとは完全に別物
└─ コネクションC (client3)  → 専用の動的テーブルC
```

- **プロセス共有ではない**: 1プロセスが何百接続捌いても、テーブルは接続ごとに独立。
  client1 が登録した cookie を client2 が index 参照しても引けない。
- **リクエストごとでもない**: 同じ接続の中なら複数リクエストをまたいで保持される。
  1回登録すれば2回目以降のリクエストで index 参照で使い回せる(= HPACK の圧縮効率の源泉)。

### なぜ接続ごとか

エンコーダ(送信側)とデコーダ(受信側)が**同じ表を同期して持つ**必要があるから。
「index 62 = あの cookie」という対応はその2者間(=1接続)だけの取り決め。
他接続と混ぜると index がズレて壊れるし、他クライアントのヘッダーが見えてしまう。

### コード上の実体

`decode` に渡す `Vec<(String, String)>` がその接続専用のテーブル。
実サーバーなら「接続ごとに `thread::spawn` した、そのスレッドのローカル変数」に相当。
接続が閉じてスレッドが終われば `Vec` も Drop されて解放。共有しないので Mutex も不要。

### 現実はサイズ上限あり(デモは簡略化)

- 最大サイズ `SETTINGS_HEADER_TABLE_SIZE`(デフォルト **4096バイト**)を接続開始時に合意。
- 各エントリのサイズ = `名前 + 値 + 32`(32はRFCの管理オーバーヘッド)。
- 上限超過で**古いエントリから追い出す(FIFO eviction)**。

### HTTP/2 Bomb との接続

「接続ごと・リクエストまたぎで保持・接続が生きてる間は解放されない」性質が足場になる。
種を1回登録 → 同じ接続で大量に index 参照 → **Slowloris で接続を閉じさせない** →
テーブルも復元メモリも接続寿命に縛られて解放されない。閉じれば解放されるのに閉じさせないのが肝。

## Q4: TCP接続はどうやって終わる? どうやって特定のクライアントだと判別している?

### 判別 = 4タプル

TCPは **(送信元IP, 送信元ポート, 宛先IP, 宛先ポート)** の4つの組で接続を一意に識別する。
カーネルがパケットごとにこれを見て「どのソケット宛か」振り分ける(デマルチプレクス)。
`accept()` のたびに接続専用のソケット(FD)が1個生まれる。

- **同じIPでも別接続なら区別できる**(送信元ポートが違う)。1人が2接続張れば
  4タプルが違う=別ソケット=**別の動的テーブル**。だから動的テーブルは「人ごと」ではなく
  「この4タプルのソケットごと」。
- コード: Rust `stream.peer_addr()` / Go `conn.RemoteAddr()` が相手の `IP:ポート`。
- **TCPの識別 ≠ ユーザー識別**: TCPは「どの回線か」しか知らない。「ログイン中の誰か」は
  HTTPのCookie/セッション/認証トークン(アプリ層)でやる別物。

### 終わり方は3つ

| 種類 | 仕組み | コード上の発火 |
|---|---|---|
| ① 正常終了 | **FIN** の4ウェイハンドシェイク | ソケットを close / drop |
| ② 強制終了 | **RST**(リセット)で即切断 | クラッシュ、強制クローズ、相手不在 |
| ③ タイムアウト | 無通信で自動切断 | `*_timeout` 設定 |

- ① net-02 の `Connection: close` は「送ったら閉じる」HTTP層の予告。実切断はその後の close→FIN。
  Rust は `drop(stream)`、Go は `conn.Close()`。
- ③ が **Slowloris の急所**: 攻撃者が timeout 直前に1バイト(WINDOW_UPDATE 等)を送り続けると、
  サーバーは「まだ生きてる」と判断して切れず、接続もテーブルもメモリも残り続ける。
  → 緩和策に「タイムアウトを厳しく」「1接続あたり保持メモリに上限」が並ぶ理由。

### HTTP/2 では Connection: close は使わない

`Connection` ヘッダーは HTTP/2 では送信禁止(RFC 9113 forbidden header)。
代わりに **`GOAWAY` フレーム**で「もう新しいストリームは受けない、畳む」と伝える。
悪用接続をサーバー側から切るときも GOAWAY / RST で畳む。

### アプリが Connection: close を明示する場面はレア

HTTP/1.1 はデフォルト keep-alive。close が付くのはほぼインフラ/フレームワークの自動判断:

- ① グレースフルシャットダウン/デプロイ(Go `srv.Shutdown` / `SetKeepAlivesEnabled(false)`)← 最頻
- ② エラー・プロトコルズレで再利用不可(Go: **ボディを読み切らずに return すると接続が閉じる**)
- ③ リソース上限(1接続あたり最大リクエスト数・アイドルタイムアウト・最大寿命)
- ④ LB/プロキシのリバランス
- ⑤ 長さを「接続クローズ」で示す古い形式(close 自体が終端マーカー)

HTTP/2 ではこれらの「接続を畳む」役割を GOAWAY が担う。
