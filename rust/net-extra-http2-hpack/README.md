# net-extra (番外編): HTTP/2 のヘッダーをバイナリで読む (HPACK) と HTTP/2 Bomb

> このトピックはネットワークカリキュラムの本編連番 (`net-01`〜`net-11`) には含まれない**番外編**。
> `net-04` は TLS ハンドシェイク用に予約されている (`NETWORK_CURRICULUM.md` 参照)。
> HTTP/2 / HPACK と、2026年6月公開の HTTP/2 Bomb を深掘りする独立トピック。

## 概要

`net-02` では HTTP/1.1 を「改行区切りのテキスト」として手書きパースした。
このトピックは **HTTP/2 のヘッダーがなぜテキストで読めないのか**、そして
**ヘッダーがバイナリでどう読まれているのか** を、クレートを使わず素のバイト列を
自分で組み立て / 復元することで1バイト単位で確認する。

そのうえで、その仕組みがそのまま脆弱性になった **HTTP/2 Bomb (CVE-2026-49975,
2026年6月公開)** を実際に再現する。

**最大のポイント:**
> HTTP/2 のヘッダーは「圧縮されたバイナリ」。一度送ったヘッダーは表に登録され、
> 以降は **数値インデックス** で参照される。この圧縮効率がそのまま増幅攻撃の燃料になる。

## なぜ HTTP/1.1 のようにテキストで読めないのか

HTTP/1.1 のヘッダーはこうだった (net-02):

```
GET /hello HTTP/1.1\r\n
Host: 127.0.0.1:7878\r\n
User-Agent: curl/8.0\r\n
\r\n
```

人間が読める。`\r\n` で区切って `split_once(':')` すれば終わり。
ところが HTTP/2 で同じヘッダーを送ると、ワイヤ上はこうなる (概念):

```
0x82 0x84 0x86 0x41 0x8c ...   ← これだけ。テキストではない
```

理由は2つ:

1. **バイナリフレーミング** — HTTP/2 はすべてを「フレーム」という固定長ヘッダー付きの
   バイナリ単位で送る。`HEADERS` フレーム、`DATA` フレーム、`WINDOW_UPDATE` フレーム…。
   1本の TCP 接続に複数リクエスト (ストリーム) を多重化するため、テキストの行区切りでは
   表現できない。
2. **HPACK 圧縮 (RFC 7541)** — ヘッダーは平文では送られず、後述の「テーブル参照」に
   圧縮される。`0x82` の1バイトが `:method: GET` を意味する、といった具合。

このトピックでは 2 の **HPACK** に絞って中身を開く。

## HPACK の3つの構成要素

### ① 静的テーブル (固定の61個)

クライアントとサーバーは、最初から **61個のよく使うヘッダー表** を共有している
(RFC 7541 Appendix A)。一部:

| index | 名前 | 値 |
|---|---|---|
| 2 | `:method` | `GET` |
| 4 | `:path` | `/` |
| 7 | `:scheme` | `https` |
| 8 | `:status` | `200` |
| 32 | `cookie` | (名前のみ) |

`:method: GET` を送りたければ「index 2」と言うだけ。**送信は1バイト**。
HTTP/1.1 ならテキストで11バイト必要だったものが、87% 削減される。

### ② 動的テーブル (接続ごとに育つ)

静的テーブルに無いヘッダー (例: 長い `cookie` の値、`user-agent`) は、
**一度だけ実バイトで送り、同時に動的テーブルへ登録**する。登録されると index 62 以降が
割り当てられ、**2回目からは index 参照 (1バイト)** で使い回せる。

これが HTTP/2 の圧縮効率の正体であり、**HTTP/2 Bomb が悪用する一点**でもある。

### ③ 可変長整数エンコード (§5.1)

index や長さの数値は「先頭バイトの下位 N ビット」に詰める。入りきらなければ
後続バイトに7ビットずつ分割し、最上位ビット `0x80` を「まだ続く」フラグにする。

```
index 5    → [0x85]              (1バイト)
index 62   → [0xbe]              (1バイト)   ← 127未満は全部1バイト
index 127  → [0xff, 0x00]        (2バイト)
index 1337 → [0xff, 0xba, 0x09]  (3バイト)
```

**index 62 までは必ず1バイト**で参照できる、という性質が後で効いてくる。

## バイナリの読み方 (実際のバイトを開く)

`cargo run` の パート1 が出力する `:method: GET` の1バイトを開くとこうなる:

```
0x82  =  1000 0010
         │   └────── 下位7ビット = 0000010 = index 2 (= :method: GET)
         └────────── 先頭ビット 1 = "Indexed Header Field" (テーブル参照)
```

HPACK は **先頭ビットを見て表現形式を判定する**:

| 先頭ビット | 表現形式 | 意味 |
|---|---|---|
| `1xxxxxxx` | Indexed (§6.1) | テーブルの index 番をまるごと参照 |
| `01xxxxxx` | Literal + 動的テーブル登録 (§6.2.1) | 新しいヘッダーを送り、表にも追加 |
| `0000xxxx` | Literal (登録なし) (§6.2.2) | 1回限りのヘッダー |

`src/main.rs` の `decode()` がまさにこの判定をやっている。
`b & 0x80` で先頭ビット、`b & 0x40` で2番目のビットを見て分岐する。
これが「サーバーがヘッダーをバイナリで読む」処理の本体。

## Goとの比較

Go では `golang.org/x/net/http2/hpack` がこのエンコード / デコードを担う。
普段は `net/http` がHTTP/2を透過的に扱うので意識しないが、中身はこうなっている:

```go
// Go: hpack デコーダ。Emit コールバックでヘッダーが1個ずつ復元される
dec := hpack.NewDecoder(4096, func(hf hpack.HeaderField) {
    fmt.Printf("%s: %s\n", hf.Name, hf.Value)
})
dec.Write(wireBytes) // ← バイナリを渡すと表参照を解決して復元
```

| 概念 | Go (`x/net/http2/hpack`) | Rust (このプロジェクト) |
|---|---|---|
| 静的テーブル | `staticTable` (内部) | `STATIC_TABLE` 定数 |
| 動的テーブル | `dynamicTable` 構造体 | `Vec<(String, String)>` |
| デコード | `Decoder.Write` + `Emit` | `decode()` 関数 |
| 整数エンコード | `appendVarInt` | `encode_integer()` |

このトピックでは「中で何が起きているか」を見るために全部手書きする。
net-02 で `net/http` を使わず `TcpListener` から手書きしたのと同じ方針。

## コード解説

| 関数 | 役割 |
|---|---|
| `encode_integer` / `decode_integer` | HPACK 可変長整数 (§5.1) の変換 |
| `encode_indexed` | Indexed Header Field (§6.1)。1バイト参照を作る |
| `encode_literal_new_name` | Literal + 動的テーブル登録 (§6.2.1)。表に種をまく |
| `table_lookup` | index → ヘッダー。1〜61=静的、62〜=動的 |
| `decode` | **受信側 (サーバー) の処理**。先頭ビットで分岐し、復元バイト数も測る |
| `dump_bytes` | バイト列を 16進・2進・ASCII で可視化 |

## 動作確認

```bash
cd rust
cargo run -p net-extra-http2-hpack
```

> 注: このリポジトリの Rust 環境は devcontainer。コンテナ内で上記を実行する。

出力は4パート構成:

1. **静的テーブル参照** — `:method: GET` が1バイト (`0x82`) になる様子
2. **可変長整数** — index の大きさでバイト数がどう変わるか
3. **動的テーブル増幅** — HTTP/2 Bomb の核心 (下記)
4. **32GB枯渇への外挿** — Slowloris との組み合わせ

## HTTP/2 Bomb (CVE-2026-49975)

> 2026年6月2日公開。発見: Quang Luong 氏 (検証: Jun Rong / Duc Phan 氏)。
> nginx / Apache / IIS / Envoy / Cloudflare Pingora のデフォルト HTTP/2 設定が対象。
> Shodan で約88万台が露出確認。

### 何がすごいのか

**家庭用100Mbps回線・1台のPC**から、サーバーに **数十GBのメモリを20秒前後で確保させ**、
応答不能にできる。実測:

| サーバー | 増幅率 | メモリ枯渇 |
|---|---|---|
| Envoy 1.37.2 | 約 5,700:1 | 32GB / 約10秒 |
| Apache httpd 2.4.67 | 約 4,000:1 | 32GB / 約18秒 |
| nginx 1.29.7 | 約 70:1 | 32GB / 約45秒 |
| IIS (Win Server 2025) | 約 68:1 | 64GB / 約45秒 |

### 仕組み = 「増幅」×「保持」

このトピックで作った HPACK の仕組みがそのまま攻撃になる。

**① HPACK 増幅 (compression bomb)** — `src/main.rs` のパート3が再現する部分。

1. 大きなヘッダー (例: 4KB の `cookie`) を **1回だけ** 送り、動的テーブル index 62 に登録
2. 以降、index 62 への **1バイト参照を数千個** 送りつける
3. サーバーは参照のたびに 4KB を **メモリ上に復元** する
   → 送信1バイト : 復元4,102バイト ≈ **4,000:1 の増幅**

Apache と Envoy が突出して重いのは、参照1回あたり約4,000バイトを確保する実装だったため。

**② Slowloris 風の保持** — メモリを解放させない仕掛け。

通常はリクエスト処理が終わればメモリは解放される。そこで攻撃者は **フロー制御
ウィンドウを0** にしてレスポンス完了をブロックし、**1バイトの `WINDOW_UPDATE`** を
定期送信して接続を生かし続ける。→ 確保されたメモリが **解放されないまま固定** される。

```
増幅 (HPACK)  ×  保持 (Slowloris)  =  数十GBのメモリ枯渇 → サーバー停止
```

### なぜ既存の防御をすり抜けるのか

- **デコード後ヘッダーサイズ上限が効かない**: メモリ確保はサイズ判定 *より前* に起きる
- **Cookie 分割バイパス**: RFC 9113 が認める「Cookie を複数フィールドに分割」を悪用。
  Apache / Envoy は分割された各 crumb をヘッダー個数にカウントしておらず、
  個数制限内で数千回の確保を達成できた

### パッチ状況 (公開時点)

| 製品 | 状況 |
|---|---|
| nginx | 1.29.8 で修正 (`max_headers` ディレクティブ追加、デフォルト1,000) |
| Apache httpd | mod_http2 v2.0.41 で修正 |
| IIS / Envoy / Cloudflare Pingora | **未パッチ** (HTTP/2 の一時無効化を推奨) |

### 緩和策

- パッチ適用済みバージョンへアップグレード
- パッチが無い場合は HTTP/2 を一時無効化 (HTTP/1.1 へフォールバック)
- ヘッダー **個数** の厳格な上限 (分割 Cookie も数える)
- ワーカープロセスへの **メモリ上限 (cgroup 等)** で巻き添えを限定

### HTTP/2 DoS の系譜 (文脈)

共通テーマは「**送信コストとサーバー側処理コストの非対称性**」。

- **2019** 一連の HTTP/2 DoS (Data Dribble, Ping Flood, Settings Flood) — フレーム制御の悪用
- **2023** Rapid Reset (CVE-2023-44487) — ストリームを開いて即 `RST_STREAM`、上限を回避
- **2025** MadeYouReset — サーバー側に RST を誘発させて Rapid Reset 対策を回避
- **2026** HTTP/2 Bomb (本トピック) — HPACK 増幅 × Slowloris 保持の **メモリ枯渇**

## 演習

### 演習1: 基礎 — 増幅率を動かす

`src/main.rs` の `big_value`(値のサイズ)と `n_refs`(参照数)を変えて、
増幅率と復元バイト数がどう動くか観察する。

- `big_value` を 64KB にすると増幅率は?
- `n_refs` を半分にすると復元バイトは?

### 演習2: 応用 — 防御策「ヘッダー個数の上限」を実装

`decode()` に `max_headers` 引数を足し、復元したヘッダー数が上限を超えたら
エラーにする。実際の nginx 1.29.8 のパッチに相当する。

ヒント: 戻り値を `Result<(Vec<(String, String)>, usize), String>` に変える。

### 演習3: チャレンジ — Cookie 分割バイパスを再現

1個の `cookie` を複数フィールドに分けて送ると、「個数」上限をすり抜けられる。
`encode_literal_new_name("cookie", ...)` を複数回呼び、個数制限が
なぜ無力になるかをデコーダ側で確認する。

## まとめ

- HTTP/2 のヘッダーは **バイナリ (HPACK)**。テキストの行区切りでは読めない
- **静的テーブル61個 + 動的テーブル** を共有し、ヘッダーを **数値インデックス** で参照する
- 先頭ビットで表現形式 (Indexed / Literal) を判定するのが「バイナリで読む」処理の本体
- index 62 までは **1バイト** で参照できる — この圧縮効率がそのまま増幅攻撃の燃料になる
- **HTTP/2 Bomb** = HPACK 増幅 (1バイト→4KB) × Slowloris 保持 (解放させない) のメモリ枯渇
- HTTP/2 の DoS は「送信コストと処理コストの非対称性」を突く系譜として理解できる
