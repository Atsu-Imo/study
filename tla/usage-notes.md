# TLA+ メモ — 使い所と、上手く使うには

`state-machine-demo/` で TLA+ を一通り触った上での、実務で使うための要点メモ。
（同じ題材の Rocq 版は `../coq/state-machine-safety/`）

---

## エコシステム早見表（言語 vs ツールを混同しない）

| 名前 | 正体 | 役割 |
|---|---|---|
| **TLA+** | 言語 | 仕様を書く記法（`.tla` の中身）。下地の数学 |
| **PlusCal** | 言語 | コードっぽく書ける上位言語。**TLA+ に変換**される |
| **TLC** | ツール | モデル検査器。到達状態を**全探索**して不変条件を検査（今回使った） |
| **Apalache** | ツール | SMT ベースの別の検査器。大きめ/記号的に強い（要型注釈） |
| **TLAPS** | ツール | 本物の証明を書く証明器（普通は使わない） |

合言葉: **「TLA+ で書いて、TLC で走らせる」**。`.tla`=モデル本体、`.cfg`=TLC への検査指示（何を・どの定数値で検査するか）。

---

## 使い所（When）

### 向いている
- **状態/並行が絡み、バグると重大**なサブシステムの**設計検証**。
  - 認可（権限）モデル：「付与されてない権限でアクセスできない」「権限昇格経路が無い」「自分の申請を自分で承認できない」
  - ワークフロー状態機械：記事/注文ライフサイクル（「承認なしに公開されない」等）
  - お金・クォータ：「二重課金しない」「残高が負にならない」
  - ロック/キャッシュ無効化/分散プロトコル：**並行インターリーブの競合**を全探索で暴く（人間が一番見落とす所）
- 確認したい性質:
  - **safety**（絶対起きてはいけない）＝「bad state に到達しない」
  - **liveness**（いつか必ず起きる）＝「提出された記事はいつかレビューされる」

### 向いていない
- CRUD、UI、フレームワーク配線、アプリ全体。→ 型・テスト・レビューで十分。
- 「全入力を数学的に完全保証」したい純粋ロジックの核。→ それは Rocq（定理証明）の領分。

### 正しさ検証のコスパ階段（下から試す）
```
1. 型で不正状態を表現不能に（enum＝直和で illegal states unrepresentable）  ← 最安・必須
2. プロパティベーステスト（ランダム入力で不変条件を大量検査）              ← 安い
3. TLA+ / モデル検査（設計に bad state が無いか・並行で壊れないか）        ← 中
4. Rocq / 定理証明（クリティカルな純粋ロジックを全入力で完全保証）         ← 高
```
ワークフロー/認可の**設計**検証なら 3(TLA+) が実務的。手証明が要らず、**反例トレース**が出るのが強み。

### TLA+ か Rocq か
- 「設計に穴が無いか素早く・並行込みで」→ **TLA+**（自動探索・反例トレース・軽い／ただし有限範囲のみ）
- 「全入力を数学的に完全保証」→ **Rocq**（無限カバー／ただし手証明が重い）

## 検証“以外”の使い道（TLA+）

「不変条件を証明/検査する」だけが TLA+ の価値ではない。むしろ副次効果が実務で効く。

- **思考の道具（最重要・過小評価されがち）**：スペックを書く行為そのものが設計の曖昧さ・抜けを炙り出す。**TLC で検査する前**にバグが見つかることが多い。Lamport 自身「TLA+ の主目的は正確に考えること」と言っている。
- **正確な設計ドキュメント**：散文の設計書と違い曖昧さゼロ。`.tla` が「生きた設計仕様」として残り、設計議論の共通言語になる。
- **バグ発見器として使う**：「証明」目的でなく「反例トレースが欲しい」だけの使い方。並行・分散の競合（人間が一番見落とす所）を全インターリーブ探索で暴く。
- **リファインメント検査**：抽象スペックと詳細スペックを2階層で書き、「詳細が抽象を正しく実装している」を検査。多段設計や、設計を段階的に詳細化するときに使う。
- **trace validation（実装↔仕様の橋）**：Go 実装に状態遷移トレースを吐かせ、それがスペックの正当なふるまいか TLC で照合 → 実装の逸脱を機械検出（AWS の手法）。モデル↔実装のズレ対策。
- **代替案の比較・設計探索**：複数の設計案をそれぞれモデル化して、どれが不変条件を保つ/壊すかを TLC で素早く比較する“設計の実験場”。

> まとめると TLA+ は「正しさを検査する」だけでなく、**曖昧さを潰して正確に考える/書く/比較する/実装を見張る**道具でもある。

## 上手く使うには（How）

1. **コードではなく設計をモデル化する。** Go を行単位で写さない。本質（状態・遷移・不変条件）だけ残して大胆に捨てる。
   - struct/DB → 変数・集合・関数(マップ)、ハンドラ → アクション、`if guard` → ガード、goroutine → アクションのインターリーブ、エラー処理/シリアライズ → 捨てる。
2. **対象を1つに絞る。** アプリ全体ではなく「壊れたら重大 × 状態/並行が複雑」な急所だけ。
3. **3点を見極める**（これがモデリングの本体・最難関）:
   - 状態変数（何のデータが効くか）
   - アクション（状態を変える操作＝主要ハンドラ）
   - 不変条件（safety＝何が起きてはダメか / liveness＝何が必ず起きるか）
   - 判断基準: 「この詳細を消すと、確認したい性質が判定できなくなる？」がNoなら捨てる。
4. **モデルは小さく。** TLC は**有限の状態空間しか見ない**ので、最初は最小規模（例: 注文1〜3件）で回す。状態爆発に注意。
5. **「有限範囲のみ」を忘れない。** 「区切った範囲で網羅」であって「全入力の証明」ではない。範囲外で出るバグは見逃しうる → 何件まで検査したかを記録。
6. **モデル↔実装のズレを管理する**（最大の実務リスク。TLA+ は設計を保証するが実装の追従は別問題）:
   - スペックを**生きた設計ドキュメント**として維持（設計変更時に必ず更新）。
   - **trace validation**: Go に状態遷移トレースを吐かせ、それがスペックの正当なふるまいか TLC で照合 → 実装の逸脱を機械検出（AWS の手法）。
7. **入口の選択**: コードっぽく書きたいなら PlusCal、素の TLA+ が分かりやすければ直書きでOK（どちらも最終的に TLC で検査）。エディタは **VS Code の TLA+ 拡張**が手軽。
8. **典型ハマり**: `UNCHANGED <<...>>` の書き忘れ（書かない変数は「任意の値」と解釈され、嘘の反例や見逃しの原因）。

## 最小ワークフロー
1. 急所サブシステムを1つ選ぶ
2. 状態変数・アクション・不変条件を書き出す
3. TLA+(or PlusCal) で抽象して書く → TLC で検査（反例が出たら設計を直す）
4. 直した設計を**手で**実装に反映、スペックは設計の正として残す
5. 必要なら trace validation で実装の追従を継続チェック
6. 「Go→TLA+ 自動変換」は期待しない（実用ツールは無い）
